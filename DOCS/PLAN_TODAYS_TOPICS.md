# 今日の話題（AI秘書 ニュース・興味トピック配信）計画書

**概要**: AI秘書が「その日始まる前の朝のニュース」と「その日の終わり頃の興味トピック調査レポート」を限定した形でお知らせする機能の計画。  
**ステータス**: 計画のみ（未実装）

**※ 朝のニュース形式（2026年3月実装）**: 朝の「本日のニューストピックス」は **RSS テキスト** から **YouTube 動画形式** に切り替え済み。指定プレイリスト・チャンネルから YouTube Data API v3 で最新動画を取得し、チャット内の小窓で埋め込み再生。再生終了で次のおすすめ動画を自動再生。API キー未設定・取得 0 件時は従来の RSS 本文にフォールバック。詳細は `includes/today_topics_youtube_helper.php` および `includes/chat/scripts.php` の動画ブロック処理を参照。

---

## 1. 目的とスコープ

- **朝（6 時〜7 時）**: **「本日のニューストピックス」**として、**分野別**（政治・経済・国際・スポーツ・IT など）にニュースを配信する。**登録ユーザー全員**に配信する。配信時刻は**その人にとって最適な時間**を AI 秘書が聞いて決める（6 時〜7 時の範囲）。**デフォルトは 7 時**。
- **夜（その日が終わる頃）**: そのユーザーの**興味が向いていそうな話題**をこちらから調査し、短いレポートとして報告する。

**配信の位置づけ**: 今日の話題（朝・夜）は**既存の「毎日1回の自動話しかけ」に統合して組み込む**。別枠の専用メッセージではなく、自動話しかけの流れのなかで「本日のニューストピックス」「興味トピックレポート」として配信する。

---

## 2. 現状の関連仕組み

- **毎日1回の自動話しかけ**: [cron/ai_proactive_daily.php](cron/ai_proactive_daily.php) が毎時実行され、`user_ai_settings.proactive_message_enabled = 1` かつ `proactive_message_hour` が現在時刻と一致するユーザーに対して、[includes/ai_proactive_helper.php](includes/ai_proactive_helper.php) の `generateProactiveMessage` で挨拶文を生成し、`ai_conversations` に `is_proactive = 1` で保存する。
- **コンテキスト**: `collectUserContext` で直近メッセージ・タスク・メモ・AI記憶を収集し、それを元に「ユーザーが興味を持ちそうな話題」で挨拶している（ニュース取得は現状なし）。

## 3. 朝の「本日のニューストピックス」（6 時〜7 時配信）

### 3.1 仕様イメージ

- **実行タイミング**: **6 時〜7 時**の枠で、ユーザーごとに設定された「最適な時刻」に配信する。**デフォルトは 7 時**。cron は 6 時と 7 時に実行し、それぞれの時刻に割り当てられたユーザーに送る（例: `today_topics_morning_hour` が 6 のユーザーに 6 時、7 または未設定のユーザーに 7 時に配信）。
- **最適な時刻の聞き取り**: **AI 秘書がユーザーに「本日のニューストピックスは何時頃にお届けするとよいですか？（6 時〜7 時の間でお選びください。指定がなければ 7 時にお届けします）」のように聞く**。会話または設定画面で回答してもらい、`user_ai_settings` の **today_topics_morning_hour**（6 または 7、デフォルト 7）に保存する。未回答・未設定の場合は 7 時とする。
- **配信対象**: **登録ユーザー全員**を想定する。自動話しかけに統合するため、**各ユーザーの today_topics_morning_hour の時刻に**、挨拶文＋本日のニューストピックスを含む 1 通を送る。今日の話題を OFF にしているユーザーは除外する。
- **内容**: **分野別**にトピックスをまとめた「本日のニューストピックス」とする。
  - 分野の例: 政治、経済、国際、スポーツ、IT・科学、社会 など（RSS/API のカテゴリに合わせて定義）。
  - 各分野ごとに見出し＋1 行要約を 1〜2 本程度、全体で 5〜10 本程度に整理する。
  - **ソーシャル・動画のトレンド**: **Twitter（X）や YouTube などで人気急上昇している話題**も取り上げる。**推奨**: ニュースと並べて**「トレンド」として 1 ブロック**を設ける（分野の一つとして含める形でも可）。**動画の場合はタイトルに加え、簡単な内容の要約**（1〜2 文）を付記する。
  - **ランキング形式**: トレンドや人気記事・動画は **ランキング形式**で提示する案とする（例: 「本日のトレンド TOP5」「人気急上昇動画 1位〜3位」「話題のツイート ベスト3」）。ユーザーが一目で把握しやすくする。
- **年代による調整**: 朝の配信は**年代別**に行う。ユーザーの年齢（10代〜70代）に応じて、**本文の内容・媒体の割合・トレンドの分量・要約の長さ**を変える（セクション 8「年代別の話題・媒体とパーソナライズ」参照）。例: 若年層は SNS・動画トレンドを多め・要約は短く、シニアはテレビ・新聞に近い信頼ソースを主軸に・トレンドは少なめ。
- **配信形態**: 今日の話題は**自動話しかけに統合**するため、既存の自動話しかけメッセージのなかに「本日のニューストピックス」ブロック（および夜は「興味トピックレポート」）を含める形とする。DB では `ai_conversations` に保存し、種別は question の固定文や種別フラグで識別する。
- **興味の案内文**: 配信文の末尾に次のような一文を必ず含める。  
  **「興味のある分野や話題があれば、次回以降で反映させていくので希望をお知らせください。」**  
  → ユーザーが返信で希望を伝えた内容を記録し、セクション 3.6 に従って次回以降のニュース選定・夜の興味レポートに反映する。

### 3.2 ニュース取得の手段（検討）

- **RSS / 公式API のみ**: スクレイピングは行わず、RSS または公式 API に限定する。
- **登録必須サイトは除外**: **会員登録・ログインしないと記事の続きが読めないサイトは情報収集候補から除外する**。RSS で取得したリンク先が有料壁・登録壁の場合も、そのソースは候補から外すか、見出しのみ利用するなど運用で対応する。

### 3.3 実装の流れ（案）

1. **ニュース取得モジュール**（新規）  
   - 朝 **6 時・7 時**の各実行で、その時刻に割り当てられたユーザー（today_topics_morning_hour = その時刻）に配信する。  
   - 取得元（RSS/API）からヘッドラインを取得し、**分野別**に分類する（RSS の category、ソースごとの分野マッピングなど）。登録必須サイトは候補から除外したソースのみ使用する。  
   - **トレンド取得**: Twitter（X）のトレンド・YouTube の人気急上昇動画などは、各公式 API から取得する（セクション 7.2 参照）。**トレンドは 1 ブロックとして**まとめることを推奨。動画はタイトル・サムネイルに加え、**説明文やメタデータから簡単な内容要約**（1〜2 文）を生成して付記する。  
   - **年代別の本文**: ユーザーごとに**年代帯**（users.birth_date から算出）に応じた内容を生成する。年代帯ごとに「ニュース本数・トレンドの割合・要約の長さ」を変え、同一の取得結果から複数パターンの本文を組み立てる。  
   - 出力は「分野 → 見出し＋短い要約」の配列に加え、「トレンド・ランキング」用の配列（TOP5 など）。全体でニュース 5〜10 本＋トレンド数件程度に整理し、年代別に文言を調整する。
2. **文言生成**  
   - 上記を AI（Gemini）に渡し、「本日のニューストピックス」という見出しと、**分野ごと**および**ランキング形式のトレンド**（「1位: … 2位: …」など）を読みやすい短文に整える。動画項目には必ず簡単な内容の要約を含める。**年代帯ごとに**プロンプトまたはパラメータを変え、トレンド多め・要約短め（若年層）／ニュース厚め・トレンド少なめ（シニア）などを反映する。  
   - **末尾に必ず追加**: 「興味のある分野や話題があれば、次回以降で反映させていくので希望をお知らせください。」
3. **一斉配信（自動話しかけに統合）**  
   - 登録ユーザー全員を対象に、**各ユーザーの年代帯に応じた本文**で 1 ユーザーあたり 1 通ずつ `ai_conversations` に保存する。**自動話しかけの同一メッセージ内**に「本日のニューストピックス」として組み込む形とする（例: 7 時に自動話しかけが実行されるユーザーには、挨拶文＋本日のニューストピックスを 1 つの answer にまとめて保存）。question は `（本日のニューストピックス）` や種別で識別する。  
   - チャット画面で「本日のニューストピックス」として表示されるように、既存の履歴表示を拡張する。

### 3.4 クリック記録で学習するアルゴリズム

**目的**: 複数のニュースを表示したとき、ユーザーが「詳細を見たい」とクリックした項目を記録し、次回以降のニュース選定・表示順・関連トピックの深掘りに反映する。

**流れ**:

1. **表示**: 朝の「今日の話題」で、各ニュースに見出し・短い要約と「詳細を見る」用のリンクまたはアクション（例: ニュースID付きのリンク／ボタン）を付与する。
2. **記録**: ユーザーがそのリンク・ボタンをクリックしたとき、以下を記録する。
   - ユーザーID、ニュースID（またはトピック識別子）、クリック日時。
   - 可能なら「どのカテゴリ・キーワードのニュースか」をメタデータとして保存する（RSS の category、見出しからのキーワード抽出など）。
3. **蓄積**: 上記を DB に保存する（例: 新規テーブル `today_topic_clicks`。カラム: user_id, topic_id / news_id, source, category_or_keywords, clicked_at）。
4. **次回以降の反映**:
   - 朝のニュース選定時: 過去のクリック履歴から「このユーザーが関心を持ちそうなカテゴリ・キーワード」を重み付けし、同じソースから取得した候補の**並び順**や**優先表示**に反映する（例: 政治・経済・スポーツのどれを多めに出すか）。
   - 夜の興味レポート: クリック履歴を `collectUserContext` と同様の「興味の手がかり」として利用し、キーワード抽出の入力に含める。
5. **バランス**: クリック履歴だけに偏らないよう、「トピックス級の重要ニュース」は毎日一定数は必ず含め、クリック学習は「その中での優先度」や「追加で1本」に使う設計とする（下記「意見の偏りを防ぐ設計」と組み合わせる）。

**DB 案**:  
- テーブル `today_topic_clicks`: id, user_id, topic_id（または external_url / 記事識別子）, source（RSS名など）, category_or_keywords（JSON または文字列）, clicked_at。  
- 集計時は直近 30 日〜90 日程度を対象にし、古いデータは削除またはアーカイブする運用を検討。

### 3.5 意見の偏りを防ぐ設計（反対意見・多様な視点）

**方針**: ユーザーが興味を示したテーマについて、**同じ立場の情報だけ**が続かないようにし、**反対意見・別の視点**の情報もきちんと届ける。

**具体的なルール**:

1. **興味テーマの識別**: クリック記録や会話から「関心が高いテーマ」（例: 特定の政策・業界・人物・論点）を抽出する。
2. **多様な視点の取得**: そのテーマについて、
   - 賛成・推進の論調の記事だけでなく、
   - 批判・慎重論・反対の論調の記事、
   - 中立的な解説・まとめ  
   を**意図的に検索・選ぶ**。キーワードに「批判」「反対」「慎重」「論争」「賛否」などを含めた検索や、複数ソースから異なる論調を1本ずつ選ぶロジックを入れる。
3. **表示の仕方**: 夜の興味レポートや、同じテーマを再度扱うときは、
   - 「〇〇について、こういう見方もあります」「別の角度ではこういう指摘もあります」のように、**反対意見・異なる視点**を必ず 1 本以上含めて要約して伝える。
4. **重み付けの上限**: クリック学習で「この論調ばかり」にならないよう、同一カテゴリ・同一論調の記事が占める割合に上限を設ける（例: 朝 5 本のうち、ユーザー好みの論調は 2 本まで、残りは多様なソース・論調から選ぶ）。

### 3.6 ユーザーが希望した興味の記録と反映

**目的**: 「興味のある分野や話題があれば、次回以降で反映させていくので希望をお知らせください。」に対してユーザーが返信した内容を記録し、次回以降のニュース選定・夜の興味レポートに生かす。

**流れ**:

1. **記録のトリガー**: 直前に配信したメッセージが「本日のニューストピックス」である会話スレッドで、ユーザーが送信したメッセージを対象にする。AI が「興味・希望」を抽出する（例: 「経済とスポーツが知りたい」「〇〇という話題をフォローして」）。
2. **保存**: 抽出した「分野・キーワード・話題」を DB に保存する。  
   - 案: 新規テーブル `user_topic_interests`（user_id, interest_type（分野 or キーワード）, value, source（message_id / 手動登録）, created_at）。  
   - または既存の AI 記憶（long-term memory）に「ユーザーが希望しているニュースの分野・話題」として書き込む。
3. **反映**:  
   - **朝のニューストピックス**: ユーザーごとの配信はしないが、夜の興味レポートや「詳細を見る」クリックとの組み合わせで、将来的にパーソナライズする場合はこの希望を重み付けに使う。  
   - **夜の興味レポート**: `collectUserContext` に「ユーザーが希望した分野・話題」を必ず含め、キーワード抽出の入力として利用する。希望に沿ったトピックを優先してレポートに含める。
4. **運用**: 希望の重複・古い希望の扱い（例: 6 ヶ月前の希望は優先度を下げる）は設計で決める。

### 3.7 「推し」の登録と有料枠での反映

**目的**: ユーザーに**推し**（応援している人物・芸能人・VTuber・アーティストなど）がいる場合、**有料枠**の夜の興味レポートで、その人に関する**最新情報**や**ファンの声**をトピックに含め、パーソナライズを強化する。

**流れ**:

1. **登録**: ユーザーが設定画面または会話で「推し」を登録する（名前・芸名・グループ名など）。AI 秘書が「推しの方を登録すると、夕方のレポートでその方の最新情報やファンの反応をお届けできます（有料プランでご利用いただけます）」と案内する。
2. **保存**: 推しの名前・識別子を DB に保存する。`user_topic_interests` の **interest_type = `oshi`** および **value = 名前（または公式アカウント名など）** とするか、専用テーブル `user_oshi`（user_id, oshi_name, oshi_type（任意）, created_at）を用意する。有料プラン加入者かつ夜の個別配信対象のみ、このデータをレポート生成に使う。
3. **反映（有料枠のみ）**: 月額プラン加入者で、かつ推しが登録されているユーザーに対して、夜の個別パーソナライズレポートに**推しに関するブロック**を 1 つ含める。内容は**その人の最新情報**（出演・リリース・SNS 発言・ニュース）と**ファンの声**（SNS 上の反応・トレンド・ファンコミュニティの話題）を検索・要約し、「〇〇さんについて、こんな動きやファンの反応がありました」という形で伝える。情報源は公式アカウント・ニュースRSS・Twitter（X）等のパブリックな投稿・トレンドに限定し、利用規約を守る。
4. **コスト**: 推し検索は 1 ユーザーあたり 1 回の追加検索・要約となるため、既存の有料枠の Gemini 呼び出しに含めて賄う。月額料金の範囲内とする。

## 4. 夜の「興味トピック調査レポート」

### 4.1 仕様イメージ

- **お試し期間（最初の 1 週間）**: 夜の興味レポートは**初週のみ**、個人の好みを抽出して**個別配信**する（Gemini でキーワード抽出＋パーソナライズ要約）。これにより利用者は体験したうえで、2 週間以降の有料継続を判断できる。
- **配信しない条件**: **昨日の分（前回配信分）でクリックしていない日が 2 日続いた人**には、その日以降の夜のレポートを**配信しない**。クリック履歴（`today_topic_clicks`）で「前日・前々日のいずれかでクリックがあるか」を判定し、2 日連続でクリックが無いユーザーは対象から外す。これでお試し期間中の配信数とコストを抑える。
- **2 週間以降の月額制**: **2 週間以降もニュース配信（夜の興味レポート）を希望する場合**は、**月額料金**を設定してコストを賄う。案内文で「お試し期間は 1 週間です。2 週間以降も夕方の興味レポートをご希望の場合は、月額 ○○円のニュース配信プランへのご加入が必要です」と明示する。
- **有料未加入者への夜の配信**: **月額を支払わない人**に対しても夜のニュース配信を行う場合は、**朝と同様にコストのかからない一斉配信**とする。すなわち、ユーザーごとの Gemini 呼び出しは行わず、1 回または年代別で数回生成した共通コンテンツを全員に配信する。加入者には個別パーソナライズ、非加入者には一斉配信（朝の「本日のニューストピックス」に近い形式で夕方にも 1 通）とすることで、全員に夕方のニュースを届けつつコストを抑える。
- **200 名超で有料に切り替え**: **利用者（登録ユーザー）が 200 名を超えた時点**で、上記の月額制に切り替える。200 名以下の間はお試しのみ（初週個別配信＋2 日連続未クリックは配信しない）とし、201 名目以降から「2 週間以降は有料」を適用する。
- **実行タイミング**: **16 時〜20 時**の間に、検索・API 負荷が集中しないよう**分散して**配信する（後述の分散実行）。1 日 1 回、対象ユーザーごとに 1 通ずつ。
- **内容**: そのユーザーの**興味が向いていそうな話題**をこちらから調査し、短いレポートとして報告する。
  - **興味の推定元**: 直近の会話・メモ・タスク・AI記憶（既存の `collectUserContext`）に加え、**ユーザーが希望として伝えた分野・話題**（セクション 3.6）および**詳細クリック履歴**（3.4）を必ず反映する。
  - **推しがいる場合（有料枠）**: 月額プラン加入者で**推し**が登録されている場合は、その人に関する**最新情報**（出演・リリース・SNS・ニュース）と**ファンの声**（SNS 上の反応・トレンド）をトピックの一つとして必ず含める。セクション 3.7 に従い、推しブロックを 1 つ追加してレポートに乗せる。
  - 調査手段: キーワードを抽出し、Gemini の Google Search grounding や、ニュースRSS/API で「そのキーワードに関連する今日のニュース」を検索。**Twitter（X）・YouTube のトレンド**も話題の一つとして含めてよい（人気急上昇のツイート・動画をランキング形式で紹介する場合、動画には簡単な内容の要約を付記する）。
  - 出力: 2〜3 トピック程度を「〇〇に興味がありそうなので、こんな話題がありました」という形で 1 メッセージにまとめる。推しがいる場合は推しブロック＋その他 1〜2 トピックとする。
- **案内の表示**: お試し期間中および 2 週間前に、AI 秘書または設定画面から「2 週間以降も夕方の興味レポートをご利用いただくには、月額 ○○円のニュース配信プランが必要です。ご希望の場合は設定からお申し込みください」と案内する。
- **負荷分散**: 夜はユーザーごとに Gemini 呼び出しとニュース検索が発生するため、16〜20 時の枠でユーザーを時間帯またはバッチに割り振る（セクション 9.1・9.4 参照）。お試しは最大 200 名×7 日かつ「2 日連続未クリック」で対象が減るため、負荷は限定的。

### 4.2 実装の流れ（案）

1. **興味キーワードの抽出**  
   - `collectUserContext` の結果に加え、**ユーザーが希望した分野・話題**（3.6）と**クリック履歴**（3.4）を入力に含める。**有料枠**の場合は**推し**（3.7）が登録されていれば、その名前をキーワードに必ず含める。  
   - これらを Gemini に渡し、「今日のニュースでフォローすると良さそうなキーワード」を 2〜3 個抽出する。
2. **検索・要約**  
   - 抽出したキーワードでニュースを検索（RSS/API または Gemini の Search grounding）。**推しが登録されている有料ユーザー**には、その人の名前・芸名で**最新情報**（ニュース・公式SNS・出演情報）と**ファンの声**（Twitter（X）等のパブリックな反応・トレンド）を検索し、1 ブロック分の要約を生成してレポートに含める。  
   - **Twitter（X）・YouTube のトレンド**を話題に含める場合は、公式 API で取得し、動画には簡単な内容の要約を付記する。ランキング形式で提示する場合は順位を明示する。  
   - 各トピックを 1〜2 文で要約し、「興味トピックレポート」用の短文を生成する。  
   - **意見の偏り防止**: セクション 3.5 に従い、興味テーマごとに反対意見・別視点の記事を少なくとも 1 本含め、「こういう見方もあります」と要約して伝える。
3. **保存・表示**  
   - `ai_conversations` に保存（例: `question = '（興味トピックレポート）'` や種別フラグ）。  
   - チャット画面で既存の自動メッセージと同様に表示する。
- **配信対象の絞り込み（お試し期間）**: 当日の夜レポートを送る前に、**「昨日の分でクリックしていない日が 2 日続いている」**かどうかを `today_topic_clicks` で判定する。前日・前々日の両方で「前回配信（朝のニューストピックスまたは前日の夜レポート）に対するクリックが 0」のユーザーは、その日の夜の配信対象から外す。

### 4.3 プライバシー・スコープ

- 興味の推定は**自サービス内のデータ**（会話・メモ・タスク・AI記憶）に限定する。  
- 外部に送るのは「キーワード」と「検索結果の要約」に留め、会話の生データを外部 API に渡さない設計とする。

## 5. 設定・オプション

- **ユーザーごとの ON/OFF**  
  - 「今日の話題」（朝ニュース）と「興味トピックレポート」（夜）をそれぞれ有効/無効にできる設定とする。  
  - 既存の `user_ai_settings` に **today_topics_morning_enabled** / **today_topics_evening_enabled**（2 本）を追加する。get_settings API で返し、設定画面で切り替え可能にする。
- **希望時刻**  
  - **朝**: **6 時〜7 時**の範囲で、**AI 秘書が「何時にお届けするとよいか」を聞き**、回答を **today_topics_morning_hour**（6 または 7）に保存する。**デフォルトは 7 時**。未設定なら 7 時で配信。  
  - **夜**: **16〜20 時の枠内**で、ユーザーごとに割り当てた時刻（またはバッチ）で配信する。希望時刻を設定できる場合は、その時刻に近いスロットに割り当てる。
- **夜のニュース配信の月額料金（2 週間以降・200 名超時）**: 利用者が **200 名を超えた時点**で、2 週間以降の夜の**個別パーソナライズ**興味レポートは**有料**とする。月額料金（例: 300〜500 円/月）を設定し、加入者には個別配信、**非加入者には朝同様の一斉配信**（コストは朝と同程度）で夕方のニュースを届ける。料金は個別配信分のコストを賄う水準にする（セクション 12.6 の再計算参照）。設定画面・決済連携（Stripe 等）は別途用意。案内は 4.1 のとおり AI 秘書・設定画面で行う。

## 7. 情報収集元の候補サイト・API

**方針**: **RSS または公式 API** に限定する。**会員登録・ログインしないと記事の続きが読めないサイトは候補から除外する**。以下は登録不要で読める範囲を前提にした候補であり、導入前に各社の利用規約・商用利用の可否・レート制限を確認すること。

### 7.1 総合・トピックス向け（朝の「本日のニューストピックス」向き）

**日本・国際（登録不要で読める範囲）**  
- **Yahoo!ニュース RSS**: 国内・経済・国際・スポーツ・IT・科学・地域など。**本機能の取得元の一つ**。  
- **CNN.co.jp**: 国際ニュースのRSS。世界中の記事を日本語で配信。**本機能で国際カテゴリの取得元として採用**。  
- **NHK**: リンク先が NHK ONE（news.web.nhk）となり登録必須のため**使用・引用しない**。RSS 取得結果に含まれる NHK リンクも `isLinkFromNhK()` で除外する。  
- **Google News（RSS）**: 非公式URLのため安定性・規約要確認。  

**海外（登録不要で読める範囲のメディアを優先）**  
- **Reuters（ロイター）**: 国際・経済のRSS。日本語版あり。多くの記事が登録不要。  
- **BBC News**: 英国放送協会。RSS あり。英語だが国際ニュースとして多様な視点の補足に有用。  
- **CNN**: 主要記事のRSS。英語。登録不要で読める記事が多い。  
- **The Guardian**: RSS あり。英語。多くの記事が無料で読める。  
- **AP News（Associated Press）**: 通信社。RSS/API。英語。中立的な報道。  
- **NPR**: 米国公共ラジオ。RSS あり。英語。  
- **Al Jazeera English**: RSS あり。多様な地域の視点を補足するのに有用。  
- **PBS NewsHour**: RSS あり。英語。  

**注意**: 朝日新聞デジタル・読売・日経などは記事の多くが**登録・有料**のため、**候補からは除外**する。RSS のみ提供でリンク先が登録壁の場合は、見出しのみ利用するかそのソースを外す。

### 7.2 ソーシャル・動画（人気急上昇トレンド）向き

- **Twitter（X）**: 公式 API（Twitter API v2）でトレンド取得が可能（例: trends/place で地域別トレンド）。有料プラン・利用規約・レート制限の確認が必要。トレンドワードや話題のツイートを「話題のツイート ベスト3」のようなランキング形式で配信に含める。
- **YouTube**: **YouTube Data API v3** で人気動画を取得可能（`videos.list` で `chart=mostPopular`、または `search.list` でトレンドに近い並びを取得）。タイトル・説明文・チャンネル名を取得し、**動画については説明文やメタデータから簡単な内容の要約**（1〜2 文）を生成して付記する。ランキング形式（「人気急上昇動画 1位〜3位」など）で提示する。
- **TikTok**: 公式 API でトレンド取得が可能な場合あり。利用規約・開発者向け制限を確認する。

**運用上の注意**: 動画・ソーシャルは著作権・利用規約に配慮し、タイトル・短い要約・公式リンクの提示に留める。ランキングの順位は API の返す指標（再生数・エンゲージメント等）に基づき、日次で更新する。

### 7.3 キーワード検索・興味レポート向き

- **Google News 検索RSS**: キーワード検索結果のRSS（非公式）。興味キーワード・反対意見の検索に利用。規約要確認。  
- **NewsAPI.org**: ニュース記事検索API。キーワード・ソース指定可。無料枠あり。日本・海外ソースを選べる。取得した記事URLの先が登録必須の場合は表示を避ける運用とする。  
- **Gemini Search grounding**: 既存検討の「キーワードでニュース検索」に利用。料金・回数制限を確認。
- **推し向け（有料枠の夜レポート）**: 推しの**最新情報**は上記のニュース検索・Google Search grounding に「推しの名前・芸名」をキーワードとして渡して取得。**ファンの声**は Twitter（X）のトレンド・検索 API でその名前が含まれるツイートやトレンドを取得し、要約して伝える。公式アカウント・パブリックな投稿に限定し、利用規約を遵守する。

### 7.4 選定時の確認事項

- **登録必須の除外**: リンク先を開いたときに会員登録・ログインを求められるサイトは候補に含めない。  
- **利用規約**: 個人利用／商用利用の可否、RSS の再配信・要約の可否。  
- **分野別の取得**: トピックスを分野別に配信するため、RSS/API でカテゴリ（政治・経済・国際・スポーツ等）が取れるソースを優先する。  
- **論調の多様性**: 反対意見を伝えるため、単一メディアに頼らず、国内外の複数ソースを組み合わせる設計を推奨。
- **トレンド・動画**: Twitter（X）・YouTube は公式 API に限定し、動画には簡単な内容の要約を付記する。ランキング形式で提示する場合は API の指標に基づき順位を付ける。

## 8. 年代別の話題・媒体とパーソナライズ（10代〜70代）

**方針**: ユーザーの**年齢**によって、提供する**情報の媒体（ニュース源・SNS・動画）**と**話題の種類・トーン**を変える。これにより、各世代が普段触れているメディアに沿った形で「本日のニューストピックス」と夜の興味レポートを届ける。

**年齢の取得**: 既存の `users.birth_date`（生年月日）から年齢を算出し、**年代帯**（10代／20代／30代／40代／50代／60代／70代以上）にマッピングする。未設定の場合は「指定なし」として、全媒体・全話題をバランスよく配信するか、設定促しを検討する。

### 8.1 年代別のメディア・情報源の傾向（検証に基づく目安）

| 年代 | 主な情報源 | 補足 |
|------|------------|------|
| **10代** | SNS（Instagram・TikTok・YouTube）、動画・画像中心 | ニュースは SNS 経由が多く、パッシブ接触。テレビ・新聞は低い。音楽・ゲーム・ファッション・勉強アプリも利用。 |
| **20代** | SNS（X が特に高い）、YouTube、Web・アプリ | X でニュース収集する割合が高い。トレンド志向。Instagram・TikTok も利用。 |
| **30代** | SNS と Web・アプリの併用 | 生活情報は SNS 約7割。ニュースは SNS 経由も多い。テレビはやや低め。 |
| **40代** | Web・アプリが最多、テレビが次点 | 生活情報は Web・アプリ約5割。テレビ視聴が 30 代より増加。 |
| **50代** | テレビ優位、Web・新聞 | テレビが最も高い。新聞・Web も利用。信頼は「テレビ」「新聞」を重視。 |
| **60代** | テレビが圧倒的、インターネット約7割が利用 | テレビ平日 4 時間強・休日 5 時間強。新聞は 20 分未満。健康・医療への関心が特に高い（約 86%）。 |
| **70代以上** | テレビ、新聞、一部でインターネット | テレビを最も信頼。情報の真偽判断では「メディアのブランド」を重視。 |

### 8.2 年代別に重視する媒体・話題の設計

- **10代**: **Instagram・TikTok・YouTube** のトレンドを多めに。話題はエンタメ・音楽・ゲーム・ファッション・勉強・部活に近いキーワードも含める。ニュースは短く・ビジュアル寄りの要約。**note**（記事プラットフォーム）や**ピクシブ**など若年層に人気のプラットフォームが API を提供する場合は候補に含める。
- **20代**: **X（Twitter）のトレンド**を必須に。YouTube 人気動画、Web ニュースのトレンド。話題は就職・お金・トレンド・社会問題まで幅広く。
- **30代**: SNS と Web ニュースのバランス。仕事・子育て・暮らしに役立つ話題、時事を要約した形で提供。
- **40代**: Web・アプリ系ニュースを中心に、テレビで扱われがちな大きなニュースも要約で補完。仕事・健康・教育・経済の話題を厚めに。
- **50代**: **テレビ・新聞で報じられるようなニュース**（NHK・主要通信社・信頼性の高い Web）を優先。健康・医療・年金・暮らしの話題を明示的に含める。
- **60代・70代**: **テレビ・新聞に近い信頼されたソース**を主軸に。健康・医療・地域・政治・社会を丁寧に要約。SNS トレンドは「世の中の話題」として 1 ブロックに抑え、長文より短文・見出し＋1 行要約を心がける。

### 8.3 若い世代向けの追加コンテンツ候補

- **Instagram**: 公式 API はビジネス用途に限定されるが、トレンド・ハッシュタグの話題を別経路（例: 検索 grounding）で補完する検討は可能。Z 世代では最多利用の SNS の一つ。
- **TikTok**: 公式 API でトレンド取得が可能な場合に採用。短尺動画のトレンドを「人気急上昇動画」としてランキング形式で提供（セクション 7.2 参照）。
- **note**: 長文・エッセイ型の記事プラットフォーム。若年層の読者も多い。API や RSS の有無を確認し、利用可能なら「話題の記事」として 1 件程度含める案。
- **ピクシブ（pixiv）**: イラスト・創作系。中高生にも人気。トレンドや特集が API/RSS で取得できるか要確認。取得可能なら「トレンド・カルチャー」の 1 項目として扱う案。
- **ニコニコ動画**: 国内動画プラットフォーム。ランキング RSS 等があればトレンド候補。利用規約・API を確認。
- **Spotify / 音楽トレンド**: 若年層で音楽アプリ利用率が高い。音楽チャート・話題の曲を「カルチャートレンド」として 1 項目含める場合は、公式 API や RSS の有無を調査する。

上記はいずれも**公式 API または公式 RSS に限定**し、利用規約・商用利用を確認したうえで採用する。

### 8.4 実装上の扱い

- **朝の「本日のニューストピックス」**: **年代別**に本文を生成・配信する。年代帯ごとに「トレンドブロック」の割合とニュースの見出し・要約の長さを変える（例: 10〜20 代はトレンドを多め・要約は短く、60〜70 代はニュースを厚く・トレンドは少なめ）。自動話しかけに統合するため、7 時の自動話しかけメッセージ内に本日のニューストピックスを含める。
- **夜の興味レポート**: 興味キーワードの抽出時に **年代帯**をコンテキストに含め、検索する媒体・話題の重み付けを変える（例: 10 代なら TikTok・YouTube のトレンドを優先、60 代なら健康・医療・NHK 系を優先）。
- **DB・設定**: 年代は `users.birth_date` から都度計算するか、`user_ai_settings` に `age_group`（例: `10s`, `20s`, …, `70s`）をキャッシュするかを設計で決定する。設定画面で「話題の傾向」を年代で選べるようにする案も検討する。

## 9. 技術検証・情報保存先・AIの成長の方向性

実装にあたり、**技術的な前提**・**情報の保存先の一覧**・**AI（配信内容）の成長の方向性**を整理する。

### 9.1 技術的な構成・検証事項

**実行環境・cron**

- **朝の配信**: 既存 [cron/ai_proactive_daily.php](cron/ai_proactive_daily.php) と同様に、**PHP CLI の cron** で **6 時と 7 時**の 2 回（または毎時実行のうち 6 と 7 のとき）に処理する。今日の話題は自動話しかけに統合するため、**各ユーザーの today_topics_morning_hour（デフォルト 7）の時刻に**、挨拶文＋本日のニューストピックス（年代別の本文）を 1 つのメッセージにまとめて保存する。ニュース・トレンドの取得は 1 回（6 時実行時）行い、その結果を 6 時・7 時の両方で使い回す。年代帯ごとに文言を変えた本文を生成してから、各ユーザーにその年代に応じた 1 通を INSERT する。
- **夜の興味レポート**: **16 時〜20 時**の間に、検索・Gemini 呼び出しが**集中しないよう分散**して実行する（詳細は 9.4「夜の分散実行とスケール」参照）。cron は 16 時スタートの 1 回起動で 4 時間かけてバッチ処理するか、16/17/18/19/20 時の毎時実行で「その時刻に割り当てられたユーザー」だけ処理する方式を採用する。
- **負荷**: 朝はニュース＋トレンドを **1 回取得**し、**年代帯ごと（最大 7 パターン程度）に本文を生成**してから、ユーザー数分の INSERT を行う。Gemini 呼び出しは年代帯数分（またはユーザーごとに年代を渡して 1 回で複数パターン生成するかは設計次第）。夜はユーザーごとに Gemini と検索が走るため、**分散実行とレート制限**が必須。

**ニュース・トレンド取得**

- **RSS**: PHP では **SimplePie** または `file_get_contents` ＋ `SimpleXML` で取得。複数フィードを順次取得する場合、**タイムアウト（例: 10 秒/フィード）**・**リトライ（1 回）**・**失敗時は当該ソースをスキップ**して続行する。
- **外部 API**: Twitter（X）・YouTube・NewsAPI 等は **cURL** で取得。**API キー**は `config/ai_config.php` または環境変数で管理し、リポジトリにコミットしない。**レート制限**に合わせて 1 リクエストあたりスリープを入れる。
- **キャッシュ**: 朝 7 時に取得した「本日分のニュース＋トレンド」は、**同一日の他ユーザー配信や再実行で再利用**するため、一時的に **ファイルキャッシュ**（例: `storage/cache/today_topics_YYYYMMDD.json`）または DB のキャッシュ用テーブルに保存する。有効期限は当日 24 時まで。

**Gemini API**

- 既存 [includes/gemini_helper.php](includes/gemini_helper.php) と [config/ai_config.php](config/ai_config.php) を利用。**Search grounding** を使う場合は、料金・1 日あたりのクォータを確認する。
- 朝の文言生成では「ニュース一覧＋トレンド一覧」をプロンプトに含めるため、**トークン数**が多くなりがち。要約を渡す形にし、**maxOutputTokens** を 1024 程度に設定して長文にならないようにする。
- **種別の識別**: 既存 `ai_conversations` は `question` と `is_proactive` で種別を判別している。本機能では **question** に固定文字列（例: `（本日のニューストピックス）` / `（興味トピックレポート）`）を入れ、**answered_by** に `gemini_today_topics` や `gemini_interest_report` を入れる場合は、`ai_conversations.answered_by` が ENUM のままなら `'ai'` にし、種別は **question の値**で判定する。

**クリック記録の送信**

- **新規 API**: 例 `api/today_topic_click.php`（POST）。パラメータ: `topic_id` または `url`, `source`, `category_or_keywords`（JSON 可）。認証は既存セッション・トークン。**1 リクエストで 1 クリック**を INSERT する。
- **フロント**: 本日のニューストピックス表示時、各項目に「詳細を見る」リンクまたはボタンを付け、クリック時に上記 API を呼ぶ。**リンク先 URL** はニュース元の URL をそのまま開くか、アプリ内で開くかは設計で決定する。

**エラーハンドリング**

- RSS/API 取得失敗時: 当該ソースをスキップし、取得できた分だけで配信する。**全て失敗した場合**は「本日は取得できませんでした」の短いメッセージを保存するか、配信をスキップするかを仕様で決める。
- Gemini 失敗時: 既存と同様にリトライまたはフォールバック。夜のレポートで失敗したユーザーはログに残し、次回再度試行するかは運用で決める。

### 9.4 夜の分散実行とスケール（16 時〜20 時・100 名〜数百万人）

**前提**: 2026 年 3 月時点では **100 名弱**の利用者を想定し、**数ヶ月で 1 万人〜数百万人**に増える可能性を見据える。夜の興味レポートはユーザーごとに Gemini 呼び出しとニュース検索が発生するため、**同時刻に全員を処理すると負荷が集中する**。16 時〜20 時の 4 時間で検索・API 呼び出しを**分散**させる。

**分散の方式（いずれかまたは組み合わせ）**

1. **時間帯スロット割り当て**: 対象ユーザーを N グループに分け、それぞれ 16:00 / 16:30 / 17:00 / … / 20:00 のスロットに割り当てる。  
   - 例: `user_id % スロット数` で余り 0 → 16:00、1 → 16:15、… のように **15 分刻みで 16 スロット**に振る。  
   - cron は **毎時・または 15 分ごと**に起動し、「今のスロットに割り当てられたユーザー」だけを取得して処理する。  
   - これにより、例えば 1 万人なら 16 スロットで 1 スロットあたり約 625 人。4 時間で均等に分散すれば 1 分あたり約 42 リクエスト程度に抑えられる。

2. **バッチ＋間隔**: 16 時に 1 回だけ cron を起動し、**キュー（DB のジョブテーブルや Redis）** に「今夜処理する user_id 一覧」を投入。ワーカーが **一定数（例: 50 人）ずつ** 取り出して処理し、**1 ユーザー処理ごとまたは N 件ごとにスリープ**（例: Gemini レート制限に合わせて 1 秒に 2 リクエストまで）を入れる。20 時までに全件処理しきれない場合は、翌日分に回すか、キューに残して 20 時以降も継続するかは仕様で決める。

3. **希望時刻がある場合**: `user_ai_settings` に「興味レポート希望時刻」（16〜20 時の何時頃か）を保存する場合は、その時刻に近いスロットに優先的に割り当てる。希望がなければ上記 1 の user_id ベースの割り当てにする。

**スケール時の目安**

- **100 名**: 16〜20 時に均等に割り振れば 1 スロットあたり約 25 人。1 分あたり数回の Gemini 呼び出しで十分。
- **1 万人**: 16 スロットで 1 スロットあたり約 625 人。15 分で 625 人を処理するには 1 分あたり約 42 回。Gemini のレート制限（例: 分あたり 60 リクエスト）内に収まるよう、スロット数やスリープで調整する。
- **10 万人〜数百万人**: **キュー＋複数ワーカー**にする。ワーカー数を増やしつつ、**Gemini と外部 API のレート制限**を超えないよう、全体のスロット数・並列数・スリープを設定する。DB の `interest_report_queue` のようなテーブルに「user_id, assigned_slot_at, status, processed_at」を入れ、未処理だけをワーカーが取りにいく形にすると、複数プロセスでも安全に処理できる。

**実装上のポイント**

- **「今夜の対象ユーザー」の確定**: 当日 16 時時点で、興味レポートが有効なユーザーを 1 回 SELECT し、その user_id リストをスロットまたはキューに振り分ける。17 時以降に登録したユーザーは翌日分から対象にするか、空きスロットがあれば当日追加するかは仕様で決める。
- **冪等性**: 同一ユーザーに対して「今日の興味レポート」を 2 回保存しないよう、保存前に「本日すでに 1 通保存済みか」をチェックする（既存の朝の話しかけと同様）。
- **設定**: 分散のスロット数・バッチサイズ・スリープは **設定ファイルまたは DB** で変更できるようにし、ユーザー数増加に合わせてチューニングできるようにする。

### 9.3 AI（配信内容）の成長の方向性

**入力シグナル（何を元に「成長」するか）**

1. **クリック履歴**（today_topic_clicks）: どのカテゴリ・キーワードのニュースを「詳細を見た」か。→ そのカテゴリ・キーワードの**重みを上げる**。
2. **ユーザーが言った希望**（user_topic_interests / ai_user_memories）: 「経済とスポーツが知りたい」など。→ **明示的な優先キーワード**として夜のレポートと将来的な朝の並び順に反映。
3. **推し**（user_topic_interests の interest_type = oshi または user_oshi）: **有料枠**の夜のレポートで、その人の最新情報・ファンの声を 1 ブロック含める。→ パーソナライズの付加価値として提供。
4. **年代**（users.birth_date → 年代帯）: 媒体の選び方・トレンドの割合・要約の長さを変える。**成長**というより**初期設定に近い**が、年代帯ごとの重み付けテーブルを運用で調整することで「よりその世代に合う」ように改善できる。
5. **既存コンテキスト**（collectUserContext）: 会話・タスク・メモ・AI記憶。→ 既存どおり「興味の手がかり」としてキーワード抽出の入力に含める。

**成長のメカニズム（どう反映するか）**

- **夜の興味レポート**: 上記 1〜5 をまとめて Gemini に渡し、「今日フォローすると良さそうなキーワード」を 2〜3 個抽出。そのキーワードでニュース・トレンドを検索し、要約＋反対意見 1 本を含めてレポート生成。**推し**が登録されている有料ユーザーには推しブロックを追加。→ **クリック・希望が多い分野ほど**レポートに選ばれやすくなる。
- **朝のニューストピックス（将来のパーソナライズ）**: 現状は全員同一内容でよいが、将来「ユーザーごとに並び順やトレンド本数を変える」場合は、クリック・希望・年代から**媒体別・カテゴリ別の重み**を計算し、同じ候補リストに対してソートしてから表示する。**重要ニュースは常に上位に固定**し、トレンド枠だけ重みで並べるなどのルールで偏りを防ぐ。
- **反対意見の挿入**（セクション 3.5）: 興味テーマが分かったら、そのテーマで「批判」「反対」「慎重」などのキーワードを付けて検索し、**少なくとも 1 本**を夜のレポートに含める。→ **同じ論調ばかりにならない**ようにする「負のフィードバック」として機能させる。

**上限・バランス（成長しすぎ・偏りを防ぐ）**

- **同一論調の上限**: 朝 5 本のうち「ユーザー好みの論調」は最大 2 本まで、など。パーソナライズ時もこの上限を適用する。
- **重要ニュースの最低本数**: クリック学習に関係なく、**政治・経済・国際などの重要トピックス**は毎日一定数（例: 3 本）は必ず含める。
- **古い希望の減衰**: user_topic_interests の created_at が 6 ヶ月以上前のものは、重みを半分にするなど**時間減衰**をかける。
- **クリックのみに依存しない**: クリックが 0 の新規ユーザーや、クリックが偏っているユーザーには、**年代＋デフォルトのバランス**で配信する。

**長期の方向性**

- **興味ベクトルの集約**: クリック・希望を集計し、**カテゴリ別スコア**（政治 0.2、経済 0.5、スポーツ 0.8 など）を月次で更新する案。user_ai_profile の preferred_topics や、専用テーブル `user_topic_weights`（user_id, category, weight, updated_at）に保持し、朝・夜の選定で参照する。
- **AI 記憶との統合**: 「ユーザーがニュースで興味を持っている分野」を要約して **ai_user_memories** に 1 件ずつ保存（category = topic_interest）すると、通常の会話でも秘書が「興味」を参照できる。**重複**を避けるため、user_topic_interests を正として記憶には「要約のみ」を書き、詳細は DB の興味テーブルに持つ設計がよい。
- **A/B テスト・効果測定**: クリック率・「詳細を見る」率をログで集計し、年代別・媒体別の**どの組み合わせがよく見られているか**を分析できるようにする。将来的に「トレンドを多めにした配信」と「ニュース多めの配信」をランダムに振り、クリック率を比較するなどの改善に使う。

## 12. 人数別コスト試算

本機能の運用コストを**ユーザー数（人数）**ごとに試算する。料金は 2025〜2026 年時点の目安であり、各サービスの価格改定により変動する。

### 12.0 コストは「検索」か「配信」か・どこでかかっているか

**結論: コストのほとんどは「配信」に対するもの（＝ユーザーごとに 1 通ずつレポートを届けるために、その都度 AI を呼んでいる部分）です。検索（RSS・外部 API でニュースを取得する行為）単体の課金は小さく、問題は「夜の興味レポートを 1 ユーザーあたり 1 回ずつ Gemini で生成している」ことです。**

| 種別 | 何に課金されるか | 現在の計画でのコスト |
|------|------------------|------------------------|
| **検索** | ニュース・トレンドを**取得する**行為（RSS 取得、Twitter/YouTube/NewsAPI の呼び出し） | **小さい**。朝は 1 回/日の取得のみ。RSS は無料。Twitter/YouTube は朝 1 回なら月額固定程度。夜は「ユーザーごとにキーワードで検索」すると NewsAPI 等のリクエスト数が人数分になるが、主因は下記の Gemini。 |
| **配信** | ユーザーに**届ける**行為（DB への保存・アプリでの表示） | **ほぼ無料**。INSERT とアプリの配信は自前インフラで賄える。 |
| **配信のための生成（ここが主因）** | 夜の「興味トピックレポート」を**ユーザーごとに 1 通ずつ、Gemini でパーソナライズして作る**行為 | **非常に大きい**。**1 ユーザー = 1 回/日の Gemini 呼び出し**（キーワード抽出＋要約）。1 万人なら 30 万回/月、100 万人なら 3,000 万回/月。ここが月額の大半を占める。 |
| **朝の生成** | 朝の「本日のニューストピックス」を年代別に**作る**行為 | **小さい**。年代帯数（最大 7 回/日）の Gemini のみ。人数が増えても 7 回/日のまま。 |

**まとめ**:  
- **検索**に対しての支払い: 朝の RSS・トレンド取得と、夜のニュース検索。朝は 1 回/日で済むので小さい。夜は「ユーザーごとに検索」する設計にするとリクエスト数が増えるが、試算の中心は **Gemini の従量課金**。  
- **配信**に対しての支払い: 配信そのもの（保存・表示）はほぼタダ。  
- **許容できないほどかかっている理由**: **夜の興味レポートを「全員に毎日、ユーザーごとに Gemini で 1 通ずつ生成する」**ため。人数に比例して Gemini 呼び出しが増え、結果として月額が人数にほぼ比例する。

**計画変更の方向性（コストを許容できる水準にする）**  
以下いずれか、または組み合わせで計画を変更する必要がある。

1. **夜の興味レポートを見直す（コスト削減の主軸）**
   - **案A**: 夜のレポートを**廃止**し、朝の「本日のニューストピックス」のみにする。朝は年代別でも Gemini は最大 7 回/日なので、月額は数百円〜数千円程度に収まる。
   - **案B**: 夜のレポートを**希望者のみ・または週 1〜2 回だけ**配信する。対象人数と回数を絞れば Gemini 呼び出しが減る。
   - **案C**: 夜のレポートを**Gemini を使わない方式**にする。ルールベースでキーワード抽出（クリック履歴・希望のキーワードをそのまま利用）し、ニュース検索は RSS/API で行い、**要約は固定テンプレートまたは簡易ルール**で作成。パーソナライズは弱くなるが、ユーザー数に依存しない固定コストに近づけられる。
   - **案D**: 夜のレポートを**「テンプレート数パターン」に限定**する。例: あらかじめ 10〜20 パターンのレポートを 1 日 1 回だけ Gemini で生成し、ユーザーは興味に近いパターンに自動割り当て。1 日あたりの Gemini 呼び出しは 10〜20 回で済む。

2. **朝は現状のまま**  
   - 朝のニューストピックスは年代別でも「1 日あたり最大 7 回の Gemini」で済むため、**朝のコストは人数に依存せず低い**。この部分は維持してよい。

3. **外部 API の見直し**  
   - Twitter/YouTube をやめ、朝は **RSS のみ**（NHK・Reuters 等）にすると、外部 API の月額固定費を削減できる。

上記を反映した**計画変更案**は、セクション 12.5 にまとめる。

### 12.1 コストの内訳

- **Gemini API**: 朝の年代別本文生成（年代帯数回または 1 回のバッチ）、夜の興味レポート（ユーザーあたり 1 回のキーワード抽出＋要約）。入力・出力トークンに応じて課金。
- **外部 API**: Twitter（X）・YouTube Data API・NewsAPI 等。無料枠を超えると従量課金。
- **RSS**: 通常は無料（サーバーからの取得のみ）。帯域・リクエスト数は自前サーバー負荷。
- **インフラ**: cron 実行・DB 書き込み・キャッシュストレージ。既存サーバーで賄う想定なら追加コストは小さい。大規模時はワーカー用インスタンスやキュー用 Redis 等の追加。
- **開発・保守**: 初回実装と、RSS/API 変更・規約対応・障害対応の工数（ここでは金額化しないが、人数が増えるほど運用負荷は増える）。

### 12.2 前提条件（試算）

- **朝**: ニュース＋トレンド取得 1 回/日、年代別本文生成で Gemini 呼び出し **最大 7 回/日**（年代帯数）。ユーザー数が増えても朝の Gemini 回数は増えない（同一本文を年代別に生成するだけ）。
- **夜**: ユーザーあたり **1 回/日**の Gemini 呼び出し（キーワード抽出＋レポート要約。Search grounding を使う場合はさらにトークン増）。
- **Gemini 単価（目安）**: gemini-2.0-flash で入力約 $0.075/100万トークン、出力約 $0.30/100万トークン（Google の公式料金表を要確認）。1 ユーザーあたり夜 1 回で入出力合計 約 2,000〜4,000 トークンと仮定すると、**1 ユーザーあたり約 $0.001〜0.002/日**（月 約 $0.03〜0.06/人、**約 5〜9 円/人・月** @ 150 円/ドル）。
- **Twitter API**: Basic/Pro プランで月額 $100〜$5,000 程度（**約 1.5〜75 万円** @ 150 円/ドル）。トレンド取得のみなら低いプランで足りる場合あり。**ここでは月 約 1.5 万円**と仮定（要確認）。
- **YouTube Data API**: 無料枠 10,000 クォータ/日。人気動画取得は 1 回あたり数クォータ。朝 1 回のみなら**無料枠内**の想定。
- **NewsAPI**: 無料枠 100 リクエスト/日。夜のユーザー数が多いと超過。有料は $449/月〜（**約 6.7 万円/月〜**）。**1 万ユーザー超で有料化の可能性**。

### 12.4 コスト削減の検討

- **夜の Gemini 呼び出し削減**: キーワード抽出と要約を 1 回にまとめる、キャッシュで類似ユーザーに同じレポートを流用する（プライバシー・品質とのトレードオフ）。
- **無料枠の活用**: Gemini 無料枠、YouTube 無料クォータ、NewsAPI 無料枠を超えないよう、ユーザー数に応じて「興味レポートを送るユーザー」をサンプリングする案（例: 毎日全員ではなく、週 2 回だけ全員に送る）。
- **外部 API の優先順位**: 朝のトレンドは NHK・Reuters 等の RSS のみにし、Twitter/YouTube はオプションにするなど、必須でない API を減らす。

### 12.5 計画変更案（コストを許容できる水準にする）

上記 12.0 のとおり、**夜の興味レポートを「全員に毎日・ユーザーごとに Gemini で 1 通生成」する現行案**では、人数に比例してコストが増え、許容できない水準になる。以下のいずれか（または組み合わせ）で計画を変更する。

|----|------|--------------------------------------|--------------|
| **A. 夜のレポート廃止** | 夜の「興味トピック調査レポート」をやめ、朝の「本日のニューストピックス」のみ提供する。 | 朝のみ: Gemini 7 回/日＋RSS＋（オプションで Twitter 等）。月額 **数千円〜2 万円程度**（外部 API 次第）。 | 夜のパーソナライズレポートがなくなる。 |

### 12.6 コスト再計算（お試し 1 週間＋200 名超で月額制）

**採用した計画**: 夜のレポートは**お試し期間として最初の 1 週間のみ**個人の好みを抽出して個別配信。**昨日の分でクリックしていない日が 2 日続いた人には配信しない**。**2 週間以降**もニュース配信を希望する場合は**月額料金**でコストを賄い、**利用者が 200 名を超えた時点**でその有料に切り替える。案内を表示する。**有料を支払わない人**には、夜も**朝同様の一斉配信**（年代別などで 1 日数回の Gemini のみ）で届けるため、非加入者分のコストはかからない。

**再計算の前提**

- **朝**: 変更なし。年代別で最大 7 回/日の Gemini。月額 約 300 円程度。
- **夜・お試し 1 週間**: 配信対象 = 「今日のニューストピックス／夜レポート」に対して、**前日または前々日のいずれかで 1 回以上クリックしたユーザー**のみ。2 日連続でクリックが無いユーザーには送らない。仮に**配信対象が有効ユーザーの 50%** とすると、100 名なら 50 人/日×7 日 = 350 回の Gemini。200 名なら 100 人/日×7 日 = 700 回。1 回あたり約 0.0015 ドル ≒ 0.23 円とすると、100 名で 約 800 円、200 名で 約 1,600 円の**お試し 1 週間のコスト**（夜の Gemini のみ）。外部 API（Twitter 等）は朝 1 回のみなら月 約 1.5 万円で固定。
- **夜・200 名超かつ 2 週間以降**: **加入者**にはユーザーごとの Gemini で個別配信。**非加入者**には朝と同様の一斉配信（年代別で最大 7 回/日程度）のみで、追加コストは朝と同程度（月額 約 300 円程度で済む）。加入者数に応じて発生するのは個別配信分のみ。加入者が 50 人なら 50×30 日 = 1,500 回/月 → 約 3,500 円/月。月額を 300〜500 円/人に設定すれば、50 人で 1.5〜2.5 万円の収入となり、コストを賄える。
