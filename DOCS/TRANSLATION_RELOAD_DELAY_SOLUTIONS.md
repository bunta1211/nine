# 言語を英語にして強制リロード時の翻訳遅延対策

## 現象

表示言語を英語（または他言語）に切り替え、**強制リロード**を押すと、画面が約30秒ほど固まったように感じる。  
原因は、リロード後に「3日以内の他者メッセージ」を**一括で自動翻訳**しており、APIが数十件を**1リクエストで逐次処理**しているため。

---

## 実施した対策（実装済み）

### 1. クライアント側：バッチ分割＋順次実行

- **ファイル**: `includes/chat/scripts.php`
- **内容**:
  - 自動翻訳で送るメッセージIDを **1回あたり最大12件** に制限。
  - 最初の12件だけを送り、結果を表示したあと、**残りがあれば2秒後に同じ処理を再実行**（`initAutoTranslation` を再度呼ぶ）。
- **効果**: 最初の12件が約3〜5秒で表示され、その間も画面操作可能。残りはバックグラウンドで順次翻訳される。

### 2. サーバー側：1リクエストあたりの上限

- **ファイル**: `api/translate.php`
- **内容**:
  - `auto_translate_messages` で受け取った `message_ids` を、**1リクエストあたり最大20件**（設定で 5〜50 の範囲）に `array_slice` で制限。
  - 定数 `AUTO_TRANSLATE_BATCH_SIZE` が `config/app.php` 等で定義されていればその値を使用。
- **効果**: 古いクライアントや大量ID送信時でも、1リクエストの処理時間が長くなりすぎない。

---

## その他の解決策（必要に応じて検討）

### 3. 表示中のメッセージだけ先に翻訳（ビューポート優先）

- **概要**: 画面上に表示されているメッセージ（ビューポート内）のIDだけを優先して送り、それ以外はスクロールで表示されそうになったときに翻訳する。
- **実装イメージ**:
  - `initAutoTranslation` で `messageIds` を集めるとき、各カードの `getBoundingClientRect()` でビューポート内か判定し、**ビューポート内 → 先頭** になるよう並べる。
  - または「最初の N 件」（今の仕様）を「ビューポート内の最大 N 件」に変える。
- **効果**: ユーザーが今見ている範囲が最優先で翻訳され、体感がさらに軽くなる。

### 4. サーバー側の並列処理

- **概要**: PHP 側で複数メッセージを **並列** で翻訳（例: 非同期HTTPで外部APIを複数同時に呼ぶ、またはキュー＋ワーカー）。
- **注意**: PHP は通常リクエスト内でマルチスレッドがないため、`curl_multi` や非同期HTTPクライアントで「複数リクエストを同時に送り、結果をまとめる」形になる。
- **効果**: 1バッチあたりの処理時間（例: 12件）を短縮できるが、実装コストとAPIレート制限の考慮が必要。

### 5. キャッシュの活用

- **現状**: `message_translations` に翻訳結果を保存しており、同じメッセージ・同じターゲット言語なら再翻訳しない。
- **推奨**: 本番で `message_translations` テーブルが存在し、インデックス `(message_id, target_lang)` が張られていることを確認する。言語切替後の2回目以降のリロードはキャッシュでかなり速くなる。

### 6. 自動翻訳の開始を遅らせる

- **概要**: ページ読み込み直後（1秒後）ではなく、**会話エリアがフォーカスされたとき**や**5秒経過後**に初回の自動翻訳を開始する。
- **実装イメージ**: `scheduleAutoTranslation` で `setTimeout(initAutoTranslation, 5000)` に変更する、または「翻訳開始」ボタンを用意してユーザーが押したときだけ `initAutoTranslation()` を呼ぶ。
- **効果**: 初回表示は原文のまま素早く出し、翻訳は必要なときだけ走らせられる。

### 7. 「今見ている会話だけ翻訳」オプション

- **概要**: 自動翻訳の対象を「現在表示中の会話のメッセージだけ」に限定する。
- **実装イメージ**: `autoTranslateCards` を取るとき、`document.querySelector('.message-list')` の子かつ `data-conversation-id` が現在の `conversationId` と一致するカードだけに絞る（既に会話単位で出している場合は、そのリストをさらに現在会話でフィルタ）。
- **効果**: 他会話のメッセージが多くても、開いている会話だけなら件数が減り、体感が軽くなる。

### 8. 定数でバッチサイズを変更可能に

- **概要**: クライアントの「12件」を設定で変えられるようにする。
- **実装イメージ**: `chat.php` や `scripts.php` で、PHP から `var AUTO_TRANSLATE_BATCH_SIZE = <?= defined('AUTO_TRANSLATE_BATCH_SIZE') ? (int)AUTO_TRANSLATE_BATCH_SIZE : 12 ?>;` のように渡し、JS の `autoTranslateBatchSize` に使う。
- **効果**: サーバー負荷や体感に合わせて、運用でバッチサイズを調整できる。

---

## 設定例（オプション）

`config/app.php` または `config/app.local.php` に以下を追加すると、サーバー側の1リクエストあたり上限を変更できる。

```php
// 自動翻訳 1リクエストあたりの最大メッセージ数（5〜50）。未定義時は 20
define('AUTO_TRANSLATE_BATCH_SIZE', 15);
```

## まとめ

| 対策 | 状態 | 効果 |
|------|------|------|
| クライアントで1回12件に分割 | 実装済み | 最初の十数件が数秒で表示され、固まった感じが軽減される |
| 残りは2秒ごとに次のバッチ | 実装済み | 全件がバックグラウンドで順次翻訳される |
| サーバーで1リクエスト最大20件 | 実装済み | 1リクエストの応答時間に上限がかかる |
| ビューポート優先 | 未実装 | 見えている範囲が最優先で速く出る |
| サーバー並列処理 | 未実装 | 1バッチの処理時間短縮 |
| キャッシュ確認 | 運用 | 2回目以降のリロードが速い |
| 自動翻訳開始を遅らせる | 未実装 | 初回表示を軽くできる |
| 現在会話のみ翻訳 | 未実装 | 対象件数削減で体感軽量化 |

英語切り替え＋強制リロード時の「約30秒止まる」という体感は、上記の実装済み対策でかなり改善される。さらに軽くしたい場合は、3・6・7 のいずれかを検討するとよい。
