# 今日の話題 cron のメモリ・負荷について

**結論**: 毎日「全員分を1プロセスでループ」するため、**ユーザー数が増えるとメモリが溜まり、同じような高負荷（メモリ増大）が起こる可能性があります。** 1ユーザーだけのテストでメモリが9000MB超になった場合、本番の朝・夜のcronでも同程度が1ユーザーあたりに乗り、ループで累積する可能性があります。

---

## 現状の動き

| 処理 | タイミング | 負荷の内容 |
|------|------------|------------|
| **朝** `ai_proactive_daily.php` | 6時・7時 | 1プロセスで「今日の話題」対象ユーザーを**全員ループ**。1人あたり Gemini 呼び出し（挨拶文生成）＋RSS本文組み立て＋DB保存。 |
| **夜** `ai_today_topics_evening.php` | 16〜20時 | 1プロセスで個別配信対象を**全員ループ**。1人あたり Gemini 呼び出し（興味レポート生成）＋DB保存。その後一斉配信対象に同じ内容を保存。 |

どちらも **同じ PHP プロセスが複数ユーザーを連続処理**するため、ループごとにメモリが解放されずに残り、ユーザー数が多いとメモリ使用量が増え続けます。

---

## 対策（実装済み・推奨）

1. **ループ内で大きな変数を解放**  
   `ai_proactive_daily.php` と `ai_today_topics_evening.php` のループ内で、処理後に `unset($greeting, $fullMessage, $content など)` を実行し、次のユーザー処理前にメモリをできるだけ戻す。

2. **ユーザー数が多い場合の運用**  
   - **朝**: 対象が数十人以上になる場合は、**1ユーザー1プロセス**で回す方式に変更することを推奨。  
     例: 対象 user_id 一覧を取得し、`php cron/send_today_topics_to_user.php 6` のように1人ずつ別プロセスで実行。各プロセス終了時にメモリが解放される。  
   - **夜**: 同様に、個別配信を1ユーザー1プロセスにするか、ループ内の unset と PHP の `memory_limit` 設定で様子を見る。

3. **PHP memory_limit**  
   cron 実行時の `php.ini` や `php -d memory_limit=512M cron/ai_proactive_daily.php` で上限を設けると、暴走時にすぐ失敗してログで分かりやすくなります。必要なサイズは「1ユーザー処理でどれだけ使うか」に依存します。

---

## 毎日の負荷が「毎回 9000 超」になるか

- **1ユーザーだけのテスト**で 9000MB 超なら、**1ユーザーあたりの処理（Gemini＋コンテキスト取得）がもともと重い**可能性があります。  
  → その場合、朝・夜のcronで「今日の話題」対象が1人でも、その1回の処理で同程度のメモリを使う可能性があります。
- **複数ユーザー**を同じプロセスで処理すると、ループごとにメモリが積み上がるため、**毎日、ユーザー数に応じてさらに増えていきます**。  
  → 対策として、上記のループ内 unset のほか、ユーザー数が多いときは「1ユーザー1プロセス」に切り替えると安全です。

**結論**: 毎日「同じ1プロセスで全員」なら、ユーザー数が多い日ほど高負荷になり得ます。unset と「1ユーザー1プロセス」の併用で、毎日の負荷を抑えられます。

---

## インスタンスを上げずに実行した場合（現状のまま）

**前提**: 1ユーザー処理で約9GB使うのに、インスタンスのメモリがそれより小さい（例: 1〜4GB）場合。

| 起きること | 結果 |
|------------|------|
| **OOM Kill** | メモリ不足になると Linux カーネルが PHP プロセスを強制終了する。cron が途中で落ちる。 |
| **一部だけ配信される** | 先に処理したユーザーには送れるが、その後のユーザーで落ちると、**後半のユーザーには届かない**。 |
| **スワップ** | スワップが有効ならディスクをメモリ代わりに使い、**極端に遅くなる**。タイムアウトやハングの原因になり得る。 |
| **PHP の memory_limit** | 例: 128M や 256M なら、9GB に達する前に PHP が Fatal error で止まる。**1人目で落ちる**こともあれば、unset で多少抑えられていれば2人目まで行ってから落ちることもある。 |

**まとめ**: インスタンスを上げずにそのまま回すと、**cron が途中で落ちる／エラーで止まる**可能性が高く、**全員に届かない・毎日安定して動かない**状態になりやすい。インスタンスを上げるか、メモリをあまり使わない運用（1ユーザー1プロセス＋小さいインスタンスで1件ずつ順番に実行するなど）にしないと、安定した毎日の配信は難しい。

---

## 1ユーザー約9GB から逆算したインスタンス・コスト目安

**前提**: 1ユーザー処理でメモリが約 9GB（9000MB）使う場合。

| 運用 | 必要なRAM目安 | インスタンス例（東京） | 月額目安（オンデマンド） |
|------|----------------|------------------------|----------------------------|
| **1ユーザー1プロセス**（推奨） | 9GB ピーク＋OS等で **約12〜16GB** | t3.xlarge（16GB） | **約 2.4〜2.5 万円/月** |
| 全員を1プロセスでループ | 9GB × 人数（例: 10人で90GB） | 大容量インスタンス（m5.4xlarge 等） | **月額 10 万円以上** になり得る |

- **t3.large（8GB）**: 9GB ピークには足りず OOM のリスクあり。**8GB では不足**。
- **t3.xlarge（16GB）**: 1プロセスで 9GB 使っても余裕あり。**1ユーザーずつ別プロセスで回す前提ならこれで足りる**想定。
- 料金は 2025 年頃の東京リージョン・オンデマンドの目安（1USD≒156円）。リザーブドやスポットで 2〜3 割安になる場合あり。

**結論**: 1ユーザー9GB を前提に「1ユーザー1プロセス」で回すなら、**EC2 は t3.xlarge クラス（月額 約 2.4〜2.5 万円）** を見込んでおくのが無難です。既存の Web 用インスタンスがこれより小さい場合は、cron 専用の小さいインスタンスを増やすか、1プロセスで複数ユーザーを回すなら unset でメモリを抑えつつ、実測で必要 RAM を確認してからインスタンスを決めるとよいです。
