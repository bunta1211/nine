# 不要ファイル・不要ページの選定と削除計画書

**目的**: 効率化・合理化・不具合の抑制・省エネ化の観点から、不要なファイル・ページを選定し、計画的に削除する。  
**原則**: 現行の実装・機能を一切変更せず、**同じことができること**を維持する。

---

## 1. 観点と選定基準

| 観点 | 選定の考え方 |
|------|----------------|
| **効率化** | 重複実装の統合、未使用コードの削除で保守コストを下げる |
| **合理化** | 役割が同じ・似たファイルを一本化し、迷いを減らす |
| **不具合が起きにくい構造** | 本番に不要な開発用・デバッグ用を減らし、誤実行・誤参照を防ぐ |
| **省エネ化** | 本番で読まないアセット・スクリプトを減らし、配信・実行負荷を抑える |

### 削除候補の判定基準

- **削除してよい**: 本番で参照されず、削除しても動作に影響がないと**コード・ドキュメントで確認できる**もの
- **要確認ののち削除**: 参照があるが「開発用・フォールバック用」などと明記され、代替が確立しているもの。**参照元を変更・削除する手順を明文化したうえで**削除
- **統合ののち一方削除**: 役割が重複する2ファイルのうち、正本を決め、もう一方を廃止する場合。**正本に機能を集約し、参照をすべて正本に寄せたあと**旧ファイルを削除
- **残す**: 本番で使われている、または「ai.php 500 対策」など**意図的な二系統**として残すとドキュメントに書かれているものは、計画で「残す」と明記する

## 3. 実施フェーズと手順

### Phase 1: 開発・デバッグ用の整理（影響小）

1. **削除または本番除外**
   - `debug-chat.php`: 開発で使用後、リポジトリから削除するか、少なくとも本番にデプロイしないことを DOCS に明記。
   - `admin/debug_*.php`, `admin/test_pdf_conversion.php`: 本番の admin から削除するか、IP 制限で保護。メニュー・リンクから外す。
2. **ドキュメント更新**
   - 「本番に含めないファイル一覧」を DOCS に追加（例: `DOCS/PRODUCTION_EXCLUDED_FILES.md`）。
   - `admin/DEPENDENCIES.md` に、削除したファイルを「削除済み」として記載。
3. **アセット**
   - test-runner.js / page-inspector.js の読込条件を確認し、本番で読まないことを DOCS に明記。

### Phase 2: 重複ファイルの統合

1. **cron 性格分析**
   - 正本を `cron/ai_profile_analyze.php` に統一（DEPENDENCIES と一致）。
   - `cron/ai_profile_analysis.php` のクエリ・条件で必要なものを `ai_profile_analyze.php` に取り込む。
   - 本番の cron 設定を `ai_profile_analyze.php` のみに変更。
   - `ai_profile_analysis.php` を削除。
   - `includes/DEPENDENCIES.md` を更新。
2. **通話 JS**
   - `chat-call.js` の参照元をすべて確認。
   - 参照が無い場合は `assets/js/chat-call.js` を削除。
   - 参照がある場合は、参照を `chat/call.js` に変更してから `chat-call.js` を削除。
   - `assets/js/DEPENDENCIES.md` を更新。

### Phase 3: ページの整理（運用方針に依存）

1. **ai_chat.php**
   - 運用で「AI相談室」を廃止する場合のみ、以下を実施。
   - メニュー・サイドバー・プッシュ・page-checker・test-runner の対象を `chat.php?secretary=1` に統一。
   - `ai_chat.php` を削除。
   - `ARCHITECTURE.md` および関連 DEPENDENCIES を更新。

## 4. 削除前の必須確認（チェックリスト）

各ファイル・ページを削除する前に、以下を実施する。

- [ ] **参照の有無**: `grep` でファイル名・URL・パスを検索し、参照元を一覧化する。
- [ ] **本番・開発の区別**: 本番でだけ読まないようにする場合は「削除」ではなく「本番から除外」または「読込条件の追加」とする。
- [ ] **DEPENDENCIES の更新**: 該当する `DEPENDENCIES.md`（api / includes / admin / assets 等）から削除するファイルを外すか「削除済み」と記載する。
- [ ] **ルールの更新**: `.cursor/rules/update-dependencies-docs.mdc` に従い、変更した箇所に対応する DOCS を更新する。
- [ ] **動作確認**: 削除後、該当機能（秘書モード・通話・管理画面・cron 等）が従来どおり動作することを確認する。

## 5. 計画の維持と変更

- 本計画書は **DOCS/** に配置し、削除・統合を実施するたびに「実施済み」を追記する。
- 新たに不要候補が出た場合は、本ドキュメントの「削除・統合候補一覧」に追記し、フェーズと手順を明記する。
- **原則の再確認**: いかなる削除も「現行の実装・機能を一切変更せず、同じことができること」を満たす範囲で行う。
