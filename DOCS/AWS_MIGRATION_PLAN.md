# Social9 AWS移転計画書

## 1. 概要

### 1.1 背景
- **現状**: heteml（共有サーバー）で本番運用
- **目標**: 1,000万人以上の利用者を想定したスケーラブルなインフラ
- **結論**: 大規模利用を見据える場合、AWSが適切

### 1.2 移転の目的
| 目的 | 内容 |
|------|------|
| スケーラビリティ | ユーザー数増加に応じた自動スケール |
| 可用性 | マルチAZによる高可用性 |
| パフォーマンス | CDN・キャッシュによる高速化 |
| 保守性 | インフラのコード化（IaC）による運用効率化 |

---

## 2. 現状 vs 目標アーキテクチャ

### 2.1 現状（heteml）
```
[ユーザー] → [heteml Webサーバー] → [MySQL（同一サーバー）]
                 ↓
            [uploads/]
            [tmp/sessions/]
```

**制約**:
- リソース上限（CPU・メモリ・ディスク）
- スケールアウトが困難
- リージョン固定

### 2.2 目標（AWS）
```
                    [CloudFront CDN]
                           │
[ユーザー] ────────────────┼────────────── [Route 53]
                           │
                    [ALB]  │
                      │    │
            ┌─────────┼─────────┐
            │         │         │
       [EC2/ECS]  [EC2/ECS]  [EC2/ECS]  ← Auto Scaling
            │         │         │
            └─────────┼─────────┘
                      │
            ┌─────────┼─────────┐
            │         │         │
       [RDS MySQL] [ElastiCache] [S3]
       (Multi-AZ)   (Redis)   (uploads)
```

---

## 3. 推奨AWSサービス構成

| 用途 | サービス | 備考 |
|------|----------|------|
| Webサーバー | EC2 または ECS Fargate | PHPアプリ。初期はEC2単体で開始可 |
| データベース | RDS for MySQL | Multi-AZ、自動バックアップ |
| ファイルストレージ | S3 | uploads/、メッセージ添付ファイル |
| セッション | ElastiCache Redis または DynamoDB | スケーラブルなセッション共有 |
| CDN | CloudFront | 静的ファイル・APIキャッシュ |
| DNS | Route 53 | ヘルスチェック付き |
| ロードバランサ | ALB | HTTPS終端、複数EC2への分散 |
| 証明書 | ACM | 無料SSL/TLS証明書 |
| 監視 | CloudWatch | ログ・メトリクス・アラーム |
| 秘密情報 | Secrets Manager | DB接続情報等 |

## 4. 移転フェーズ

### Phase 0: 準備（1〜2週間）
- [ ] AWSアカウント作成・組織設計
- [ ] コスト見積もり・予算承認
- [ ] 移転チーム・役割分担の決定
- [ ] 現行環境の完全バックアップ取得

### Phase 1: 開発・検証環境構築（2〜3週間）
- [ ] VPC・サブネット設計
- [ ] EC2（またはECS）でPHPアプリ動作確認
- [ ] RDS MySQL 構築・データインポート
- [ ] S3バケット作成（uploads移行用）
- [ ] 環境変数・設定のAWS対応（ Secrets Manager等）
- [ ] 動作検証・パフォーマンステスト

### Phase 2: 本番環境構築（2〜3週間）
- [ ] 本番用VPC・セキュリティグループ
- [ ] RDS Multi-AZ構築
- [ ] Auto Scaling Group 設定
- [ ] ALB・CloudFront 設定
- [ ] Route 53 ホストゾーン作成
- [ ] SSL証明書（ACM）申請・紐付け

### Phase 3: データ移行（1〜2週間）
- [ ] MySQLダンプ取得（本番停止時間最小化のため最後に）
- [ ] S3へのファイル移行（uploads/、uploads/messages/）
- [ ] DBインポート・整合性確認
- [ ] セッションのクリア（移行後は新規ログイン）

### Phase 4: カットオーバー（1日）
- [ ] 最終同期（DB・ファイル）
- [ ] DNS切替（Route 53）
- [ ] 旧環境の読み取り専用化（ロールバック用）
- [ ] 動作確認・監視

### Phase 5: 安定化・最適化（2〜4週間）
- [ ] 監視・アラート設定
- [ ] パフォーマンスチューニング
- [ ] コスト最適化
- [ ] 旧環境の段階的廃止

## 5. アプリケーション変更点（チェックリスト）

### 5.1 設定ファイル
| ファイル | 変更内容 |
|----------|----------|
| `config/database.php` | RDSホスト・認証情報を環境変数から取得 |
| `config/app.php` | `UPLOAD_DIR` を S3パスに変更（またはS3アダプター） |
| `config/session.php` | Redis/DynamoDBセッションストア対応 |

### 5.2 必要対応
- [ ] S3へのファイルアップロード対応（`api/upload.php`、`api/messages.php`のupload_file）
- [ ] セッションをRedisまたはDynamoDBに保存するよう変更
- [ ] ログ出力をCloudWatch Logs連携（任意）
- [ ] ヘルスチェック用エンドポイント（`/api/health.php` 等）の整備

### 5.3 環境変数（推奨）
```
DB_HOST, DB_NAME, DB_USER, DB_PASS
REDIS_HOST, REDIS_PORT
S3_BUCKET, S3_REGION, AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY
APP_ENV=production
```

## 6. データ移行手順

### 6.1 MySQL
```bash
# heteml上で
mysqldump -u user -p database > backup_$(date +%Y%m%d).sql

# AWS上で
mysql -h your-rds-endpoint -u admin -p database < backup_YYYYMMDD.sql
```

### 6.2 ファイル（uploads/）
```bash
# aws cli使用
aws s3 sync ./uploads/ s3://your-bucket/uploads/ --exclude "*.log"
```

### 6.3 サイズ見積もり
- DB: 現行サイズを確認し、RDSインスタンスサイズを決定
- S3: uploads/ の合計容量を確認

## 7. ロールバック計画

| 状況 | 対応 |
|------|------|
| 重大障害発生 | Route 53をhetemlに向けて即切戻し |
| 部分的な不具合 | 機能単位で切り戻し検討 |
| データ不整合 | 移行前バックアップから復元 |

**前提**: カットオーバー前にhetemlの完全バックアップを保持し、1週間は即時切戻し可能な状態を維持

## 9. コスト見積もり（月額概算）

### 9.1 1万人規模（初期）の内訳

| 項目 | 内容 | 月額（円） |
|------|------|------------|
| **EC2** | t3.micro 1台（vCPU 2、メモリ1GB） | 約1,200 |
| **RDS** | db.t3.micro（MySQL、20GB） | 約1,800 |
| **S3** | ストレージ20GB + リクエスト | 約500 |
| **Route 53** | ホストゾーン1件 | 約75 |
| **データ転送** | 1万人なら数十GB程度 | 約500〜1,000 |
| **合計** | 最小構成 | **約4,000〜5,000円/月** |

※ ALB・CloudFront は省略可能（単一EC2で開始、スケール時に追加）

### 9.2 無料枠を使う場合（初年度）

| 項目 | 無料枠 | 月額 |
|------|--------|------|
| EC2 | 750時間/月（t2.micro） | 0円 |
| RDS | 750時間/月（db.t2.micro） | 0円 |
| S3 | 5GB、リクエスト | 0円 |
| **合計** | アカウント登録から12ヶ月間 | **約0〜2,000円/月** |

※ Route 53・データ転送のわずかな費用が発生する場合あり

### 9.3 構成別の目安

| 規模 | 構成 | 月額目安 |
|------|------|----------|
| 1万人 | EC2 1台 + RDS + S3（最小） | 4,000〜6,000円 |
| 10万人 | EC2 2〜3台 + ALB + RDS + S3 | 2〜4万円 |
| 100万人 | Auto Scaling + Multi-AZ RDS + CloudFront | 10〜30万円 |
| 1,000万人 | フルスケール構成 | 50〜160万円 |

### 9.4 主な費用の内訳

- **EC2**: サーバー稼働時間 × 単価。24時間稼働で約1,200円/月（t3.micro）
- **RDS**: DB稼働時間 × 単価。24時間稼働で約1,800円/月（db.t3.micro）
- **S3**: 保存容量 + リクエスト数。数十GBなら数百円
- **ALB**: ロードバランサ利用料。複数台構成時に必要（約2,400円/月）
- **CloudFront**: データ転送量。無料枠1TB/月あり

## 10. リスクと対策

| リスク | 対策 |
|--------|------|
| ダウンタイム | 深夜帯カットオーバー、事前告知 |

| コスト超過 | 予算アラート、リザーブドインスタンス検討 |
| スキル不足 |  AWS認定資格者との連携、マネージドサービス活用 |
