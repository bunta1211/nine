# 事務局・ポーリング・画像削除 問題の解決シート

本ドキュメントは、事務局グループで発生していた問題、ポーリングによる過去ログ表示の問題、画像（メッセージ）削除が効かない問題、および**左パネル「他○件を表示」で下にスクロールできない問題**の原因と解決内容を記録したものです。

---

## 1. 事務局で発生していた問題

### 1.1 現象

- **事務局グループ**ではファイル送信・通常メッセージ送信が、送信直後は表示されるが**リロードすると消える**
- 別アカウント（KEN）やアプリグループでは同様の不具合は出ない
- 送信後に **403** が多発する事象も発生

### 1.2 原因（想定）

- **本番DBのスキーマ差**  
  環境によって以下の有無が異なる可能性があった。
  - `messages`: `message_type` / `content_type`, `deleted_at` / `is_deleted`, `message_reactions`, `message_mentions`
  - `conversation_members`: `left_at`
- 存在しないカラム・テーブルを参照する SQL が実行され、**例外**や**更新がコミットされない**状態になっていた。
- **レート制限**でポーリング等のリクエストが多くなり、**IP がブロック**され、以降 403 が返る状態になっていた。

### 1.3 解決内容

| 対象 | 対応内容 |
|------|----------|
| **api/messages.php（get）** | `message_type`/`content_type`、`deleted_at`/`is_deleted`、`message_reactions`、`message_mentions`、`conversation_members.left_at` を `SHOW COLUMNS` / `SHOW TABLES` で判定し、存在するものだけを使って SELECT/WHERE を組み立て。例外時はフォールバックで最小限の SELECT のみ実行。`Cache-Control: no-store, no-cache` を付与。 |
| **api/messages.php（send / upload_file）** | 会話メンバー確認で `left_at` を任意に。INSERT で `content_type`/`message_type` の有無を確認して種別を保存。INSERT 直後に `inTransaction()` なら `commit()` を実行。 |
| **config/database.php** | PDO 作成後に `PDO::ATTR_AUTOCOMMIT` が定義されていれば `true` を設定。 |
| **includes/interceptor.php** | チャット用 GET（messages, conversations, friends, tasks, memos, notifications, calls）は迎撃スキップ。レート制限超過時は **IP ブロックせず**、そのリクエストのみ **429** を返すように変更。 |

## 2. ポーリングで過去ログがすぐ最新に戻される問題

### 2.1 現象

- 過去のメッセージ（過去ログ）を見にスクロールしても、**すぐに最新にスクロールし直され**、過去ログを閲覧できない。

### 2.2 原因

- ポーリングで新着メッセージを追加するたびに、**無条件で** `messagesArea.scrollTop = messagesArea.scrollHeight` を実行していたため、ユーザーが過去を見ている状態でも強制的に最下部に戻されていた。

### 2.3 解決内容

| 対象 | 対応内容 |
|------|----------|
| **includes/chat/scripts.php** | 新着追加**前**に「最下部から 200px 以内にいるか」を `wasNearBottom` で判定し、**`wasNearBottom` が true のときだけ**最下部へスクロールするように変更。過去ログ閲覧中はスクロール位置を維持。 |

---

## 3. 事務局グループで画像（メッセージ）を削除しても削除されない問題

### 3.1 現象

- 事務局グループで画像（メッセージ）を削除しても、**何度削除しても削除されない**（削除後も表示されたり、リロードで再表示される）。

### 3.2 原因（想定）

- 本番の `messages` テーブルに **`deleted_at` カラムが存在しない**（または古いスキーマで `is_deleted` のみ）場合、削除 API が `UPDATE messages SET deleted_at = NOW()` のみを行っており、**SQL エラー**または**更新対象 0 件**となり、削除が反映されていなかった。
- 削除は論理削除のため、**一覧取得（get）や初期表示**で「削除済みを除外」する条件が、`deleted_at` のみを参照しており、`is_deleted` で削除したメッセージが除外されていなかった。

### 3.3 解決内容

| 対象 | 対応内容 |
|------|----------|
| **api/messages.php（delete）** | `deleted_at` / `is_deleted` の有無を判定し、存在するカラムをすべて更新（両方あれば `deleted_at = NOW(), is_deleted = 1`）。UPDATE 後に `inTransaction()` なら `commit()` を実行。どちらも無い場合は 500 と error_log。 |
| **api/messages.php（get）** | 取得時に `deleted_at` があれば `m.deleted_at IS NULL`、`is_deleted` があれば `(m.is_deleted = 0 OR m.is_deleted IS NULL)` の両方で削除済みを除外。既読更新用の MAX(id) も同条件で除外。 |
| **api/messages.php（get フォールバック）** | メインの SELECT が例外で落ちたときに使うフォールバッククエリにも、`deleted_at` / `is_deleted` の除外条件を追加。事務局などでメインクエリが失敗してフォールバックに落ちていた場合、削除済みが復活していた原因を解消。 |
| **includes/chat/data.php** | 会話一覧・メッセージ取得・既読更新で、`deleted_at` / `is_deleted` の存在を `SHOW COLUMNS` で判定し、存在する方で削除済みメッセージを除外するように変更。初期表示やリロード時も削除済みが表示されないように統一。 |

## 4. 返信の引用がリロードで消える問題（リアクション同様の対応）

### 4.1 現象

- 返信で送信したメッセージは送信直後は引用ブロックが表示されるが、**リロードすると引用が消える**。

### 4.2 原因（想定）

- **get のフォールバック**で、メインの SELECT が例外で落ちたときに使う最小限の SELECT に **reply_to_id / reply_to_content / reply_to_sender_name が含まれていなかった**。そのため、メインクエリが失敗してフォールバックに落ちた場合（スキーマ差や一時的なエラーなど）、取得したメッセージに引用情報が付かず、リロード時に引用が表示されない状態になっていた。
- リアクションでは同様にフォールバックに **reaction_details** を付与する対応をしており、引用も同じパターンで必要だった。

### 4.3 解決内容

| 対象 | 対応内容 |
|------|----------|
| **api/messages.php（get フォールバック）** | `reply_to_id` カラムの有無を `SHOW COLUMNS` で判定。存在する場合はフォールバック SELECT に `m.reply_to_id` と `LEFT JOIN messages rm, users ru` を追加して reply_to_content / reply_to_sender_name を取得。フォールバックで組み立てる各メッセージに `reply_to_id` / `reply_to_content` / `reply_to_sender_name` を付与（リアクションの reaction_details と同様）。 |
| **api/messages.php（send）** | 送信後に `UPDATE messages SET reply_to_id = ? WHERE id = ?` を実行し、リロード後も引用が表示されるように DB に保存。 |

### 4.4 解決確認

- リアクションがリロードで消えていた問題と同じ原因（get フォールバックに情報が含まれていなかった）だったため、同じ解決法（フォールバックでも reply_to_id および返信情報を取得・付与）を適用した。
- 本番に `api/messages.php` を反映したところ、**引用がリロード後も消えなくなったことを確認**（解決済み）。

## 6. 左パネル「他○件を表示」で下にスクロールできない問題

### 6.1 現象

- 左パネルで「他 91 件を表示」を押しても**下にスクロールできず**、一覧の下が見られない。
- 展開しても**すぐに折りたたみ状態に戻ってしまう**（ポーリングで会話リストが更新されるたびに「他○件を表示」に戻る）。

### 6.2 原因

- **スクロール**: 左パネル・会話リストが flex レイアウトで、子要素に `min-height: 0` が無かったため、リストが「中身の高さ」に張り付き、スクロール領域ができていなかった。親の `.left-panel` にも `min-height: 0` が必要だった。
- **展開が戻る**: 会話リストのポーリングで `applyLeftPanelFilter()` が定期実行されるたびに `show-all` / `expanded` を外していたため、ユーザーが「他○件を表示」で展開してもすぐに折りたたみに戻っていた。

### 6.3 解決内容

| 対象 | 対応内容 |
|------|----------|
| **assets/css/chat-main.css** | `.left-panel` に `min-height: 0` を追加。`.conversation-list` / `.friends-list` に `min-height: 0`、`overflow-x: hidden`、`-webkit-overflow-scrolling: touch` を指定し、展開時に会話リスト領域内で縦スクロール可能にした。 |
| **includes/chat/scripts.php** | `applyLeftPanelFilter()` で、フィルターが「友達」以外のときは `show-all` を消さないように変更。展開状態を `localStorage.convListExpanded` で保持し、ポーリング後も `convListExpanded === 'true'` なら `show-all` とフッター「少なく表示」を再適用するようにした。 |

### 6.4 解決確認

- 「他○件を表示」を押すと全件表示になり、会話リストを**下にスクロールできる**ことを確認。
- ポーリングが走っても**展開状態が維持**され、すぐに折りたたみに戻らないことを確認。

## 5. パソコン版事務局でファイルが送れない（添付ダイアログが開かない）場合

### 5.1 現象

- 事務局グループなどで ⊕ をクリックしてもファイル選択ダイアログが開かず、ファイルが送れない。

### 5.2 対応内容

| 対象 | 対応内容 |
|------|----------|
| **includes/chat/scripts.php** | 添付クリック時、`#galleryInput` / `#imageInput` を `inputArea` 内からもフォールバック取得。`mainAttachBtn` が無い・別テンプレートでも、事務局等のグループ入力エリアから確実にファイル input を開くようにした。 |

## 7. 変更ファイル一覧（反映時に送るファイル）

本対応を本番に反映する際は、少なくとも以下のファイルをアップロードしてください。

| ファイル | 主な変更 |
|----------|----------|
| `api/messages.php` | get/send/upload_file/delete の耐障害化、deleted_at / is_deleted 両対応、commit、get での is_deleted 除外。**返信引用**: get フォールバックに reply_to_id 等を付与、send で reply_to_id を UPDATE で保存。 |
| `config/database.php` | autocommit の明示 |
| `includes/interceptor.php` | チャット用 GET の迎撃スキップ、レート制限時 429 のみ（IP ブロックしない）。POST messages.php/upload.php は迎撃スキップ |
| `includes/chat/scripts.php` | ポーリング時のスクロールを「最下部付近のときだけ」に変更。事務局等で添付が開くよう input のフォールバック追加。**左パネル**: 「他○件を表示」展開状態を localStorage で維持し、applyLeftPanelFilter で上書きしないように変更。 |
| `includes/chat/data.php` | 会話一覧・メッセージ取得・既読更新の deleted_at / is_deleted 両対応 |
| `assets/css/chat-main.css` | **左パネル**: .left-panel に min-height: 0、.conversation-list / .friends-list に min-height: 0 と overflow 指定で「他○件を表示」後に下にスクロール可能に。 |

## 8. 補足

- 事務局だけ問題になる要因として、**本番 DB のスキーマ差**を前提に、存在するカラム・テーブルのみを使う耐障害化を行っている。
- 画像削除が効くようにするため、**削除 API** と **取得・一覧の両方**で `deleted_at` / `is_deleted` の両方に対応している。
- 今後のスキーマ変更（例: 本番に `deleted_at` を追加する）を行った場合でも、既存の `is_deleted` 環境と両立する実装になっている。
