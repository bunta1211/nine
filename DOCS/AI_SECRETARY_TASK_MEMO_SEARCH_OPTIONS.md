## 背景

- **記憶範囲**: タスクとメモに限定（ユーザーが明示的に保存した内容のみ）
- **削除後も記憶**: 削除しても履歴として残す（論理削除＋検索で削除済みも含む）
- **キーワード検索**: 秘書への指示で「〇〇をまとめて報告して」→ キーワードを抽出して検索
- **例**: 「2025年度の怪我をまとめて報告して」→ 「怪我」で検索し「〇件の怪我があったようです」と案内

## 案1: リアルタイム検索（DB検索＋1回のAI応答）

### 概要
ユーザーが指示したタイミングで、DBからタスク・メモを検索し、結果をAIが整形して応答する。

### フロー
1. ユーザーが「2025年度の怪我をまとめて報告して」と入力
2. 正規表現でキーワード「怪我」と期間「2025年度」を抽出
3. `tasks` と `memos` を `LIKE '%怪我%'` で検索
4. 検索結果（件数＋要約）をAIに渡し、自然な報告文を生成

### メリット
- 常に最新データを参照
- 検索ロジックが単純
- 要約用の事前バッチは不要

### デメリット
- 1回の応答で1〜2秒かかる
- レート制限（60 RPM）の影響を受けやすい
- 毎回Gemini APIを呼ぶためコストが発生

### 想定数値
- DB検索: 10〜50ms
- API呼び出し: 1回、約1,200 input / 200 output tokens
- 応答時間: 0.5〜2.5秒
- 月1万回: 約300円

## 案2: ハイブリッド（事前要約＋オンデマンド検索）

### 概要
定期的にタスク・メモの要約を保存し、検索指示時はその要約を優先して参照。詳細が必要なときだけリアルタイム検索を行う。

### フロー
1. **バックグラウンド**: cronで1日1回、ユーザーのタスク・メモを要約し `ai_task_memo_summaries` に保存
2. **検索時**: まず保存済み要約を検索。キーワードが含まれていればそれをベースにAIが応答
3. **詳細が必要な場合**: 要約に詳細が足りなければ、その時点でDBを検索して補足

### メリット
- 検索時は要約だけ読むのでトークン使用量が少ない
- 応答が速くなる場合がある
- 日次バッチなのでレート制限を分散できる

### デメリット
- 実装が複雑（新テーブル、cron、要約ロジック）
- 直近の追加・変更は反映されない
- バッチ実行分のAPIコストが発生

### 想定数値
- 日次要約: 1ユーザーあたり1回、約500 tokens/ユーザー
- 検索時: 要約参照で約800 input / 150 output tokens
- 応答時間: 0.3〜1.5秒（要約のみの場合）

---

## 案3: シンプル検索（DB検索のみ、AIは集計と案内）

### 概要
AIはキーワード抽出と最終案内のみ担当。検索はDBで完結し、集計結果を定型文で返す。

### フロー
1. ユーザーが「2025年度の怪我をまとめて報告して」と入力
2. AI（またはルール）でキーワード「怪我」と期間「2025年度」を抽出
3. DBで `tasks` と `memos` を検索し、件数と簡易一覧を取得
4. 定型テンプレートで「〇件の怪我がありました。（一覧）」と返す（AI呼び出しなし、または最小限）

### メリット
- 実装が最も簡単
- APIコストをほぼゼロにできる
- 応答が速い（100ms〜500ms）
- レート制限の影響を受けない

### デメリット
- 回答が機械的になりやすい
- 複雑な質問や要約には向かない
- 自然言語での「まとめ」が弱い

### 想定数値
- DB検索: 10〜50ms
- AI呼び出し: 0回（キーワード抽出もルールベースの場合）または1回（抽出のみ）
- 応答時間: 100〜500ms
