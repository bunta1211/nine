# 電話・通話機能の現状と高機能化案（LINE同等を目指す）

## 1. 現在の機能の整理

### 1.1 実装済みの構成

| 項目 | 内容 |
|------|------|
| **入口** | チャット画面（会話選択中）の通話ボタン → メニューで「ビデオ通話」「音声通話」を選択 |
| **バックエンド** | `api/calls.php`（create / join / leave / decline / get_active / history） |
| **DB** | `calls`（conversation_id, initiator_id, room_id, call_type, status 等）、`call_participants`（参加・招待・拒否・退出） |
| **メディア** | Jitsi Meet（meet.jit.si）をルームIDで利用。音声・ビデオ・画面共有・バーチャル背景・背景ぼかしに対応 |
| **UI** | `includes/chat/call-ui.php`（着信モーダル・通話メニュー・通話中インジケーター・ビデオウィンドウ・コントロールバー） |
| **着信** | Web Push（`push_helper.php` の `triggerCallPushNotification`）＋通知DB＋ポーリング（get_active）で着信検知。着信音・バイブ・着信モーダル（出る/拒否） |
| **別画面** | `call.php`（会話ID・タイプをクエリで受け取り、Jitsi のみの専用ページ。参加者サイドバー・通話中チャットUIあり） |

### 1.2 現状の動作フロー

1. **発信**: チャットで「ビデオ通話」または「音声通話」選択 → `api/calls.php` create → ルーム作成・参加者招待・Push 送信 → `showCallUIAndStartJitsi(roomId, isVideo)` で Jitsi 埋め込み＋自分カメラ表示。
2. **着信**: Push またはポーリングで着信検知 → 着信モーダル表示・着信音ループ → 「出る」で join → 同上で Jitsi 開始。
3. **終了**: 通話終了ボタン → Jitsi hangup・call_participants 更新・ルーム削除処理。

### 1.3 制限・未実装になっている点

- **P2P / 低遅延**: Jitsi の TURN/STUN は Jitsi 側設定に依存。自前の TURN サーバーは未設定。
- **着信の確実性**: ブラウザの Push・バックグラウンド制限により、タブを閉じていると着信に気づきにくい。
- **通話履歴の見せ方**: API（history）はあるが、チャット画面に「通話履歴」としての一覧・未着信表示が弱い。
- **LINE 的な UX**: 通話中の「通話タブ」や「通話で戻る」、グループ通話の招待追加・通話中のチャット送信と履歴連携などは未整備。
- **音声のみの最適化**: 音声通話時も Jitsi ルームは同じ。LINE のような「音声だけ」の軽い UI は call.php でビデオオフにしている程度。

## 2. LINE と同等にするためのギャップと高機能化の方向性

### 2.1 LINE の電話機能で備えている主な要素

- **音声通話**: 1対1・グループ。通知・ロック画面着信・着信音設定。
- **ビデオ通話**: 1対1・グループ。カメラ切替・背景ぼかし・画面共有。
- **通話タブ / 履歴**: 通話リスト・未着信・履歴一覧・会話ごとの通話履歴。
- **通話中の UX**: 通話中もチャットでメッセージ送受信・通話から戻るときの「通話に戻る」バナー。
- **グループ通話**: 途中参加・退出・主催者/参加者の役割（LINE はシンプルだが、招待の追加はある）。

### 2.2 ギャップ整理と対策の対応表

| ギャップ | 現状 | 高機能化の方向 |
|----------|------|----------------|
| 通話履歴の可視化 | history API のみ。チャットに組み込み弱い | 会話詳細や専用「通話」エリアに履歴一覧・未着信表示を追加 |
| 着信の確実性 | Push＋ポーリング。タブ閉じで弱い | サービスワーカー Push・アプリ化（PWA/ネイティブ）でバックグラウンド着信。必要なら VAPID 等の整備 |
| 通話中のチャット | call.php に通話中チャット UI はあるが、メッセージ API と未連携 | 通話中チャットを `api/messages.php` と連携し、同じ会話にメッセージとして保存・表示 |
| 「通話に戻る」 | 通話中に他タブへ行くと戻りにくい | 通話中は常にインジケーターを出し、「クリックで通話ウィンドウにフォーカス」または最小化バーで復帰 |
| 音声のみの軽さ | 同じ Jitsi ルームでビデオオフ | 音声通話はビデオストリームを最初から要求しない設定を明示（既存の startWithVideoMuted の徹底） |
| グループ通話の途中参加 | ルームはあるが、招待の再送・「リンクで参加」が未整備 | アクティブな通話への「参加リンク」生成・招待メッセージ送信。参加者一覧のリアルタイム更新 |
| ネットワーク品質表示 | 未実装 | Jitsi の統計または getStats で遅延・パケットロスを取得し、品質インジケーターを表示 |
| 着信音・バイブ設定 | 設定で call_ringtone あり。種類は限定的 | 着信音プリセットの拡充・カスタム着信音アップロード（要ストレージと再生権限） |

## 3. 音声通話・ビデオ通話の実装・強化案

### 3.1 共通基盤（現状維持＋小さな強化）

- **媒体**: Jitsi Meet を継続利用。自前で WebRTC スタックを組むより運用コストが低い。
- **認証**: 通話は「ログイン済みアプリ内」のみ。電話番号認証は必須にしない（現状どおり）。
- **ルームID**: 会話ベースの `social9_{conversation_id}_{timestamp}_{random}` を継続。衝突リスクは低い。
- **未成年・制限**: 既存の `approved_contacts`・`call_restriction` を維持しつつ、グループ通話の「誰を追加できるか」を将来拡張可能にしておく。

### 3.2 音声通話（LINE 同等の体験）

- **発信**: 現状の「音声通話」で create → Jitsi は `startWithVideoMuted: true` で開始。カメラは最初から取らない。
- **UI 分岐**: 音声通話時はチャット内の通話 UI を「大きなアバター＋名前＋タイマー＋マイク/スピーカー/終了」だけのシンプルレイアウトにし、ビデオパネルは出さない（または最小化）。
- **着信**: 着信モーダルで「ビデオ通話」「音声通話」を区別表示（現状の `call_type` をそのまま活用）。
- **通話品質**: 音声のみのときはビットレート・解像度を下げる設定を Jitsi で明示し、弱い回線でも切れにくくする。

### 3.3 ビデオ通話（LINE 同等の体験）

- **現状の維持**: 2 パネル（自分／相手）・マイク/カメラ/背景ぼかし/バーチャル背景/画面共有/終了はそのまま。
- **複数人**: 3 人以上で `call-multi-participant` のレイアウトを継続。必要ならタイル数・サイズを調整。
- **品質表示**: 上記の通り、遅延・パケットロスを簡易表示（「接続状態: 良好/悪い」など）。
- **通話中のチャット**: 通話中に送ったメッセージをその会話のメッセージとして保存し、チャット履歴と通話履歴の両方で見られるようにする。

### 3.4 着信・通知の強化

- **Web Push**: 既存の `triggerCallPushNotification` を維持。payload に `call_id`・`room_id`・`call_type` を含め、クリックで `call.php?c=...&type=...` またはチャットの着信 UI に飛ばす。
- **ポーリング**: チャット画面を開いているときは `get_active` の間隔を現状のまま（または少し短く）し、着信モーダル・着信音と連携。
- **PWA / アプリ**: 将来、サービスワーカーで Push を受けてバックグラウンドでも着信表示・着信音を鳴らす。その際は `call.php` またはインアプリの通話 UI を起動する URL を Push に含める。

## 4. 段階的な実装計画（案）

### Phase 1: 現状の安定化と UX の揃え（短期）

- チャット画面で「通話履歴」を会話詳細または右パネルに表示（`api/calls.php` history を利用）。未着信があれば「未応答」ラベルを付与。
- 音声通話時の UI を「音声専用」レイアウトにし、ビデオパネルを出さない。
- 着信モーダルで「〇〇さんからビデオ通話」「〇〇さんから音声通話」を明確に表示（既存の `call_type` を活用）。
- 通話終了後、必要なら「通話が終了しました」を会話にシステムメッセージとして残す（オプション）。

### Phase 2: 通話中のチャット連携（中期）

- `call.php` の通話中チャット送信を `api/messages.php` と連携し、同じ会話にメッセージとして保存。
- チャット画面の通話 UI（埋め込み Jitsi）からも通話中にメッセージ送信できるようにし、同じ API を使う。
- 通話開始/終了を会話タイムラインに「〇〇がビデオ通話を開始しました」等のシステムメッセージで残すか検討。

### Phase 3: 「通話に戻る」とグループの強化（中期）

- 通話中に他タブや別画面へ行った場合、「通話中」インジケーターを常に表示し、クリックで通話ウィンドウにフォーカスまたは call.php を開く。
- グループ通話で「参加リンク」を生成し、招待メッセージとして会話に送信。リンクから call.php で参加できるようにする。
- 参加者一覧のリアルタイム更新（Jitsi の participantJoined/Left と `get_active` または WebSocket の将来拡張）。

### Phase 4: 品質・信頼性（中長期）

- Jitsi の統計 API 等で遅延・パケットロスを取得し、通話 UI に「接続状態」を表示。
- 必要に応じて自前の TURN サーバー（例: coturn）を導入し、Jitsi または JVB の設定で指定。
- 着信音・バイブの設定を強化（プリセット追加・カスタム着信音のアップロードと再生）。

### Phase 5: アプリ・バックグラウンド着信（長期）

- PWA 化またはネイティブアプリでサービスワーカー Push を有効にし、タブを閉じていても着信通知・着信音を鳴らす。
- 着信時のアクションで「出る」を選んだときに call.php またはインアプリ通話 UI を起動するフローを統一。

## 6. まとめ

- **現状**: 音声・ビデオとも Jitsi Meet で実装済み。発信・着信・参加・拒否・終了・履歴 API があり、着信は Push＋ポーリングで対応。
- **LINE 同等にするには**: 通話履歴の可視化、通話中のチャット連携、「通話に戻る」、音声専用 UI、グループの参加リンク・品質表示・着信の確実性（PWA/アプリ）を段階的に追加する。
- **まず手を付けやすいのは**: Phase 1 の通話履歴表示と音声通話時の UI 分岐。続けて Phase 2 の通話中チャットのメッセージ連携を入れると、LINE に近い体験になる。

このドキュメントは `DOCS/` に配置し、通話機能の仕様変更時や新規タスクの参照として利用する。

## 7. 料金がかかる選択肢とクオリティの数値

クオリティを高めるために有料サービスを導入する場合の選択肢と、**数値で表せる品質向上の目安**をまとめる。

### 7.1 現状（無料）の目安

| 指標 | 現状の目安 | 備考 |
|------|------------|------|
| **接続成功率** | 約 65〜80% | STUN のみの場合。社内ネット・対称NAT・厳しいファイアウォールでは P2P が張れず失敗しやすい。 |
| **接続確立までの遅延** | 数百 ms 〜 数秒 | 公設サーバー負荷・地域によりばらつき。 |
| **通話中の RTT（往復遅延）** | 100〜400 ms 程度 | 地域・回線に依存。meet.jit.si はグローバル1拠点に依存しがち。 |
| **ビデオ解像度・ビットレート** | 720p / 約 2.5 Mbps 想定 | Jitsi の一般的な推奨値。無料枠では保証なし。 |
| **SLA・稼働保証** | なし | 障害時も保証なし。 |

※ 接続成功率の「約 35% 改善」は、適切な TURN 導入により P2P で張れない環境でもリレーで張れるようになるという研究値（STUN/TURN 最適化の文献）を参考にしている。

---

### 7.2 選択肢 A: 有料 TURN サーバーのみ導入（Jitsi は現状のまま）

**内容**: 現在の meet.jit.si はそのまま使い、**メディア経路のフォールバック用にだけ**有料 TURN を用意する。Jitsi のカスタム TURN 設定にそのサーバーを指定する。

| 項目 | 内容 |
|------|------|
| **主なサービス例** | Xirsys（$9.99/月〜25GB 等）、Twilio TURN（$0.004/分のリレー利用）、Metered TURN（約 $99/月〜150GB） |
| **想定コスト** | 月 約 $10〜100（利用量による）。720p ビデオは 1 参加者あたり約 2 Mbps のため、150GB/月 で小規模〜中規模程度。 |
| **接続成功率の改善** | **約 65〜80% → 約 90〜98%**（環境による）。対称NAT・厳しいファイアウォール越しでは TURN がないと失敗するため、TURN 導入でその分が解消される。文献では「適切な STUN/TURN で接続成功率が最大で約 35% 向上」との報告あり。 |
| **遅延** | リレー経由時は P2P より 20〜80 ms 程度増えることが多いが、**「接続できない」よりは遅延が増える方がマシ**。地域分散された TURN なら、同一リージョンでは +20〜40 ms 程度に抑えられる場合がある。 |
| **その他** | 解像度・ビットレートの保証はない（現状の Jitsi のまま）。 |

**向いているケース**: まず「つながらない」を減らしたい場合。コストを抑えつつ効果が数値で出やすい。

---

### 7.3 選択肢 B: Jitsi as a Service（8x8）に切り替え

**内容**: meet.jit.si の代わりに 8x8 の「Jitsi as a Service」を使う。API は Jitsi 互換で、インフラと SLA が付く。

| 項目 | 内容 |
|------|------|
| **料金例** | Developer: 無料（25 ユーザー）。Basic: $99/月（300 MAU）。Standard: $499/月（1.5K MAU）。Overage: $0.99/MAU。録画は $0.01/分 等。 |
| **接続成功率・品質** | エンタープライズ向けインフラのため、**TURN/STUN が整備され接続成功率は 95% 前後以上**を想定。公設 meet.jit.si より「つながらない」が減る。 |
| **稼働** | **99.99% SLA**（公式）。無料の meet.jit.si には SLA なし。 |
| **遅延** | 地域分散された 8x8 の拠点を使うため、**同一国内・同一リージョンでは RTT が 100〜250 ms 程度に収まりやすく**、公設 1 拠点より安定しやすい。 |
| **解像度・ビットレート** | 公式の「数値の保証」は検索では見つかっていないが、商用基盤のため **720p〜1080p が安定して出しやすい**環境と想定できる。 |

**向いているケース**: ユーザー数が増えてきて、安定性と SLA が欲しい場合。Jitsi のコードをあまり変えずに乗り換え可能。

---

### 7.4 選択肢 C: 他プロバイダ（Daily / Agora 等）に全面切り替え

**内容**: Jitsi ではなく、Daily.co や Agora などの WebRTC PaaS に全面切り替える。SDK ・API の書き換えが必要。

| 項目 | Daily.co | Agora |
|------|----------|--------|
| **料金目安** | ビデオ 約 $0.0015〜0.004/参加者分、音声 約 $0.00036〜0.001/参加者分。**月 1 万分無料**。 | ビデオ HD 約 $3.99/1000 参加者分、Full HD 約 $8.99/1000 参加者分。**月 1 万分無料**。最低 $500/月 の枠あり（Tier 1）。 |
| **接続成功率** | 商用 TURN/STUN とグローバル分散により **90% 後半〜99% 台**をうたう業者が多い。 | 同様に高めの接続成功率を想定。 |
| **遅延** | グローバルエッジで **RTT を 100〜200 ms 台に抑える**ことを謳うことが多い。 | 同様。 |
| **解像度・品質** | **HD 込みの単価**。720p/1080p を想定。Network Quality API 等で品質可視化がしやすい。 | **解像度別の単価**（HD / Full HD / 2K）。フル HD 以上を明示的に選べる。 |
| **開発コスト** | 現在の Jitsi 埋め込みから **SDK ・ルーム生成・UI の書き換え**が必要。 | 同様。 |

**数値的なクオリティ向上の目安**:
- 接続成功率: **約 65〜80% → 約 95〜99%**（TURN 整備・エッジ分散による）。
- 通話中 RTT: **100〜400 ms → 80〜200 ms 程度**に改善しうる（リージョン・回線による）。
- 解像度: 720p を安定させつつ、**1080p やフル HD を「選択可能」にできる**（Agora は課金 tier で明示）。

**向いているケース**: 新規に通話基盤を選ぶ、または Jitsi をやめてまで品質と機能を揃えたい場合。Twilio Video は提供終了のため、Daily / Agora / LiveKit 等が候補。

### 7.5 選択肢の比較まとめ（数値の目安）

| 選択肢 | 想定コスト（目安） | 接続成功率の目安 | 遅延の目安 | 解像度・品質 | 実装負荷 |
|--------|--------------------|------------------|------------|--------------|----------|
| **現状（無料）** | $0 | 約 65〜80% | RTT 100〜400 ms | 720p 想定・保証なし | なし |
| **A: TURN のみ** | 月 $10〜100 | 約 90〜98% | リレー時 +20〜80 ms | 現状と同程度 | 小（設定追加） |
| **B: Jitsi as a Service** | 月 $99〜499+ | 約 95% 以上 | RTT 100〜250 ms 程度 | 720p〜1080p 想定、SLA あり | 中（エンドポイント切替） |
| **C: Daily / Agora 等** | 従量 または 月 $500+ | 約 95〜99% | RTT 80〜200 ms 程度 | HD〜Full HD 選択可能 | 大（SDK 全面書き換え） |

- **「つながりやすさ」を数値で改善したい**: まず **A（TURN）** で接続成功率を約 20〜30 ポイント程度上げるのが効果的。
- **安定性と契約ベースの保証が欲しい**: **B（Jitsi as a Service）** で SLA とインフラの安定性を得る。
- **遅延・解像度まで含めて最大限クオリティを上げる**: **C（Daily / Agora 等）** で RTT 低減と 1080p 等の選択が可能になるが、開発コストは大きい。

- **遅延・解像度まで含めて最大限クオリティを上げる**: **C（Daily / Agora 等）** で RTT 低減と 1080p 等の選択が可能になるが、開発コストは大きい。

### 7.6 C案（Daily / Agora）導入時の料金シミュレーション

実際に C 案を導入したとき、**毎日の利用者数**に応じて無料枠内に収まるか・有料にどのように変遷するかを試算する。

#### 前提条件

- **課金単位**: どちらも **participant-minutes（参加者×分数）**。例: 2人で10分通話 = 20 participant-minutes。
- **1人あたり1日の通話時間**: 平均 **10分** とする（1対1・短めの通話を想定）。  
  → 厳しめの想定（1人20分/日）も別表で記載。
- **月間 participant-minutes** = 毎日の利用者数 × 10分 × 30日。
- **Daily**: 月 **10,000 分まで無料**、超過分 10,001〜100,000 分は **$0.004/分**、100,001〜500,000 分は **$0.0037/分**（ volume discount）。クレジットカード不要で無料枠利用可能。
- **Agora**: 月 **10,000 分まで無料**、超過分は **$3.99/1,000 分**（Video HD）= 約 $0.004/分。一部契約では **最低 $500/月** がかかる場合あり（Tier 1 等）。ここでは「従量のみ」と「最低 $500 あり」の2パターンを記載。

#### シミュレーション結果（1人あたり10分/日）

| 毎日の利用者数 | 月間 participant-minutes | 無料枠(1万)との関係 | Daily 月額（概算） | Agora 月額（従量のみ・概算） | Agora 月額（最低$500ありの場合） |
|----------------|---------------------------|----------------------|--------------------|------------------------------|----------------------------------|
| **3人**        | 900                       | 無料枠内             | **$0**             | **$0**                      | **$500**                         |
| **10人**       | 3,000                     | 無料枠内             | **$0**             | **$0**                      | **$500**                         |
| **100人**      | 30,000                    | 2万が有料            | 20,000 × $0.004 ＝ **約 $80**  | 20,000 × $0.00399 ≒ **約 $80** | **$500**（従量$80 < 最低のため） |
| **500人**      | 150,000                   | 14万が有料           | 90,000×$0.004 ＋ 50,000×$0.0037 ≒ **約 $585** | 140,000×$0.00399 ≒ **約 $559**（ volume 5% で約 $531） | **約 $559**（従量が最低を上回る） |

- **3人・10人**: どちらも **無料枠内**。Daily は $0。Agora は「従量のみ」なら $0、「最低 $500 あり」の契約なら **$500/月固定** になる。
- **100人**: 無料を超え有料に変遷。Daily・Agora とも従量で **約 $80/月** 程度。Agora で最低 $500 がある契約の場合は **$500/月**。
- **500人**: 有料部分が大きくなり、**約 $530〜585/月**。Daily は volume discount でやや高めでも $600 前後。

#### シミュレーション結果（1人あたり20分/日・厳しめ）

| 毎日の利用者数 | 月間 participant-minutes | Daily 月額（概算） | Agora 月額（従量のみ・概算） |
|----------------|---------------------------|--------------------|------------------------------|
| **3人**        | 1,800                     | **$0**             | **$0**                      |
| **10人**       | 6,000                     | **$0**             | **$0**                      |
| **100人**      | 60,000                    | 50,000×$0.004 ＝ **約 $200** | 50,000×$0.00399 ≒ **約 $200** |
| **500人**      | 300,000                   | 90,000×$0.004 ＋ 200,000×$0.0037 ≒ **約 $1,220** | 290,000×$0.00399×0.95 ≒ **約 $1,100** |

#### 無料→有料の変遷の目安

- **Daily**
  - 月間 **10,000 分以下** → **$0**（毎日おおよそ **33人×10分** までなら 10,000/30 ≒ 333 分/日 → 約 **33人** まで無料）。
  - 10,001 分を超えた分から **$0.004/分**。100,001 分超から **$0.0037/分** に割引。
- **Agora（従量のみの場合）**
  - 月間 **10,000 分以下** → **$0**（Daily と同様の目安）。
  - 超過分は **約 $3.99/1,000 分**。100K〜500K 分で約 5% 割引。
- **Agora（最低 $500/月 の契約の場合）**
  - 利用が少なくても **毎月 $500**。従量が $500 を超えたら従量分を請求（100人・10分/日では約 $80 のため $500 のまま。500人で従量が $500 を超えると従量に）。

※ 為替・各社の最新料金表・契約形態により変動するため、導入時は必ず公式の料金ページで再計算すること。

このセクションは料金・品質の具体的な数値は各社の最新料金表・SLA に依存するため、導入時には必ず公式の最新情報で確認すること。

## 8. 自前構築案（C案代替・従量課金ゼロ化）【確定】

C案（Daily / Agora）と同等の品質を**自前サーバー＋オープンソース**で実現し、通話の従量課金をゼロにする。以下は推奨に基づき確定した計画である。

### 8.1 目的とスコープ

- **目的**: 外部の通話 PaaS への従量課金をやめ、自前インフラで接続成功率・遅延・解像度を C 案に近づける。
- **スコープ**: 音声通話・ビデオ通話・画面共有・背景ぼかし等は現状どおり維持。録画・PSTN 等は別検討。
- **コスト**: ソフトは無料。**VPS または専用サーバー（CPU・メモリ・帯域）の費用は発生する**。規模に応じて月数千円〜数万円程度を想定。

### 8.2 採用する構成

**採用**: **8.2-B（自前 Jitsi を最初から構築）**。自前 Jitsi（JVB ＋ Jicofo ＋ Prosody）＋ 自前 TURN（coturn）を一括で構築する。meet.jit.si は使わない。

参考（採用しない案）:

| 案 | 内容 |
|----|------|
| 8.2-A | TURN のみ自前、メディアは meet.jit.si のまま。経由しない。 |
| 8.2-C | Mediasoup 等で SFU を自前実装。工数が大きいため採用しない。 |

### 8.3 技術スタック（採用）

- **TURN/STUN**: **coturn**。通話用サーバー（1 台目）に同一でインストールするか、同一 VPS 内で JVB と併存させる。ポート 3478（UDP/TCP）等を開放。
- **SFU・シグナリング**: **Jitsi Meet スタック**
  - Jitsi Videobridge（JVB）: メディア中継
  - Jicofo: カンファレンス制御
  - Prosody: XMPP（Jitsi のシグナリング）
  - 公開用: Nginx リバースプロキシ ＋ SSL（Let's Encrypt）
- **既存との接続**: `api/calls.php` で発行している `room_id` を、自前 Jitsi の「ルーム名」としてそのまま利用する。フロントは `meet.jit.si` の代わりに **自前 Jitsi の URL**（`https://meet.自社ドメイン/ルーム名`）を開くように変更する。

### 8.4 サーバー要件・コスト目安

- **通話用サーバー（1 台・日本 1 リージョン）**
  - JVB: 4 vCPU・8GB RAM 以上推奨。帯域は同時通話数と解像度に依存（720p で 10 人同時なら 数十 Mbps 程度）。
  - Prosody / Jicofo / Nginx / coturn: 同一サーバーで兼用可。
  - 目安: 月 **$50〜150**（小規模）。規模が大きい場合は JVB をスケールアウトし、月 **$200〜500** もあり得る。
- **ホスティング**: **通話専用の別 VPS または EC2**（既存 Web サーバーとは分離する）。

※ 料金は目安。実際の同時通話数・解像度・リージョンで変動する。

### 8.5 既存システムとの統合

- **api/calls.php**: `room_id` は現状どおり発行。返却する **join_url** を「自前 Jitsi のベース URL ＋ room_id」に差し替える（設定または環境変数でベース URL を指定）。
- **チャット画面（scripts.php / call-ui.php）**: `JitsiMeetExternalAPI` のドメインを **meet.jit.si** から **自前 Jitsi のドメイン**（`meet.自社ドメイン`）に変更。`chat.php` で `window.__JITSI_DOMAIN` 等を渡し、フロントで参照する。
- **call.php**: ルーム名は `room_id` のまま。ドメインのみ自前 Jitsi（`meet.自社ドメイン`）に変更する。

### 8.6 実装フェーズ

1. **フェーズ 1**: 自前 Jitsi の設計・構築。通話専用 VPS に JVB ＋ Jicofo ＋ Prosody をインストール（Docker または手動）。Nginx、ドメイン（`meet.自社ドメイン`）・SSL（Let's Encrypt）。同一サーバーに coturn を構築し、Jitsi の TURN 設定に自前 coturn を指定する。**繋がらない主因**: meet.jit.si は configOverwrite.startConference を尊重しないため会議が開始されない。自前 Jitsi では config.js で **everyoneIsModerator または startConference を保証**し、会議を即開始させること。
2. **フェーズ 2**: `api/calls.php` およびフロント（scripts.php / call-ui.php / call.php）の join_url ・Jitsi ドメインを自前 Jitsi 向けに切り替え。動作確認（発信・着信・参加・終了）。  
   **※ アプリ側の切り替えは実施済み**: `config/app.php` に `JITSI_BASE_URL` を追加。`app.local.php` で `JITSI_DOMAIN` / `JITSI_BASE_URL` を上書き可能。`api/calls.php` の join_url、`chat.php` の `window.__JITSI_DOMAIN` / `__JITSI_BASE_URL`、`includes/chat/scripts.php`・`call.php`・`assets/js/chat/call.js`・`assets/js/chat-call.js` が設定を参照。自前サーバー構築後は app.local.php でドメインを指定するだけで切り替わる。  
   **※ 終了時は必ず leave を呼ぶ**: 発信者・着信者いずれも「通話終了」またはタブ閉じ／リロード時に `api/calls.php?action=leave` を呼ぶ実装にしてあり、相手側の get_active に同じ着信が残り続けないようにしている。詳細は DOCS/CALL_CONNECT_AND_NOTIFICATION_FIX_PLAN.md。
3. **フェーズ 3**: 負荷テスト・接続成功率の計測。必要に応じて TURN を複数リージョンに配置するか検討（8.7 参照）。
4. **フェーズ 4**: 監視・ログ・障害時の手順を整備。証明書の自動更新（certbot）。

### 8.7 複数リージョン（参考・将来の拡張）

**本計画では初期は 1 リージョン（日本 1 拠点）で開始する。** 以下は、将来 TURN または JVB を複数リージョンに配置する場合の比較である。

| 観点 | 複数リージョンの利点 | 複数リージョンの欠点 |
|------|----------------------|------------------------|
| **遅延** | ユーザーに近いリージョンに接続できるため、**RTT を 20〜80 ms 程度削減**できる場合がある。遠隔地同士の通話でも、中継を「中間」に置ける。 | — |
| **接続成功率** | 1 リージョンが障害や回線混雑で使えなくても、**他リージョンにフォールバック**できる。対称NAT・厳しいFW環境でも、地理的に近い TURN の方が UDP が通りやすいことがある。 | — |
| **コスト** | — | **サーバー台数・帯域が増えるため、月額が 2〜3 倍**になりやすい。監視・設定の対象も増える。 |
| **運用** | — | **証明書・設定・ログの管理がリージョンごとに必要**。デプロイ・障害対応が複雑になる。 |
| **シンプルさ** | — | **1 リージョンの方が構成が単純**。トラブルシュートがしやすい。 |

- **採用**: 利用者が主に国内で、同時通話数がそれほど多くない段階では **1 リージョン（日本 1 拠点）で開始**する。遅延や接続失敗の実測を見てから複数リージョンを検討する。
- 海外ユーザーが多かったり、接続成功率・遅延を最優先する場合は、のちに TURN を 2 リージョン（例: 日本＋米国またはシンガポール）に置く選択肢がある。

### 8.8 その他の採用事項

以下は、推奨に基づき**採用した選択肢**である。利点・欠点は決定の理由として参照する。

#### 8.8.1 自前 Jitsi のドメイン・URL

- **採用**: **サブドメイン `meet.自社ドメイン`**（例: meet.social9.jp）。運用が分かりやすく、Jitsi の公式ドキュメントや Nginx の例もサブドメイン前提である。

| 選択肢 | 利点 | 欠点 |
|--------|------|------|
| サブドメイン `meet.自社ドメイン`（採用） | 通話専用であることが分かりやすい。Let's Encrypt で 1 証明書にまとめやすい。 | サブドメインの DNS 設定が必要。 |
| パスで分ける | ドメインを増やさず済む。 | リバースプロキシ・ルーティングがやや複雑。 |

#### 8.8.2 サーバー配置

- **採用**: **通話専用の別 VPS（または EC2 の通話専用インスタンス）**。メディア・TURN の負荷と障害を Web サーバーから分離する。既存 EC2 は Web/API 用のままとする。

| 選択肢 | 利点 | 欠点 |
|--------|------|------|
| 別 VPS（採用） | 通話専用にでき、Web と分離。障害の波及を防ぎやすい。 | 契約・管理対象が増える。 |
| 既存 EC2 に載せる | 契約・支払いが一元化できる。 | リソース競合・障害の影響範囲が Web と重なる。 |

#### 8.8.3 録画・ストリーミング

- **採用**: **初期は録画・ストリーミングは対象外**。音声・ビデオ・画面共有を安定させたのち、必要になった段階で Jibri または外部サービスを検討する。

| 選択肢 | 利点 | 欠点 |
|--------|------|------|
| やらない（採用） | 構成がシンプル。Jibri 等が不要。 | 録画・アーカイブは提供しない。 |
| 自前（Jibri 等） | 従量課金なしで録画可能。 | サーバー・ストレージ・運用負荷が大きい。 |
