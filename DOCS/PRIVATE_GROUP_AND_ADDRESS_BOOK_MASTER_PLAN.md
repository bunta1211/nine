## 3. 削除（やめること・出さないこと）

| 削除・禁止内容 | 説明 |
|----------------|------|
| チャットからのプライベートグループ作成 | `api/conversations.php` の `create` では `is_private_group` を true にしたリクエストを受け付けない。フロントの「新規グループ作成」からもプライベートグループは選べない。 |
| 「友達追加」というラベルの継続使用 | 個人向けの入口・モーダル・説明では「個人アドレス帳」に統一し、「友達追加」のラベルは使わない。 |
| 未登録者への招待で「友達申請を送る」のみの表現 | 「招待メール送信」と「〇〇のアドレス帳に追加受諾」に置き換える。 |
| 組織招待で既存ユーザーに「パスワード設定して承諾」のみ | 既存ユーザーには「統合する／キャンセル」の選択を表示し、統合案内を出す。 |

**9.2 で決定した方針の反映**:
- 左パネルフィルタ: `filter_friends` の表示を「個人アドレス帳」に統一（lang.php の ja を「個人アドレス帳」に変更）。
- 設定画面: セクション名「友だち管理」→「個人アドレス帳」に変更。
- 既存ユーザー向け組織招待: 同一 `accept_org_invite.php?token=` で、ページ内でパスワード設定済みなら統合UIを表示。
- 一斉招待: 組織のメンバー管理画面に同居させ、**画面・ナビの名前を「組織アドレス帳」に変更**（admin の「メンバー管理」→「組織アドレス帳」）。招待候補の登録・一斉招待を同じ画面で提供。
- プライベートグループ: 既存グループを後からプライベートに変更できるようにする（管理画面で is_private_group と4設定を編集可能にする。検証で問題なければ実装）。
- member_list_visible=0: 他メンバー名は非表示、**メンバー数（例: 3人）は表示する**。

## 4. 変更しないもの（現状維持）

- 友達申請の**データ構造**（friendships の request/accept）は変更しない。UIの文言だけ「アドレス追加申請」に変える。
- グループ作成の**チャット側フロー**（組織あり・なし、既存グループのクローン）は、プライベートグループを追加しない限りそのまま。
- 組織の「グループチャットを追加」（通常グループ）は従来どおり admin/groups.php と admin/api/groups.php で作成可能。
- 検索ポリシー（メール・携帯番号のみで検索等）は維持。表現だけ「アドレス追加申請」「招待メール送信」に揃える。
- `invite.php` の招待リンク・トークン方式は維持し、メール本文とラベルのみ「〇〇のアドレス帳に追加受諾」に変更。

## 6. 実装順序の目安

1. **DB**: migration_private_group_settings.sql の作成と適用（手動実行はユーザー方針に従う）。

### 9.3 検索まわり計画との関係

- 検索統合計画（SEARCH_INDEX / 検索まわり統合と仕組みの見直し）で、**友達検索（個人アドレス帳検索）**の API パラメータを `query` に統一し、文言を lang キーに寄せる方針。マスター計画の「アドレス追加申請」「招待メール送信」は、その **lang キーまたは SEARCH_INDEX の UI・文言一覧** と一致させる必要がある。
- 実装順序: 検索まわりで SEARCH_INDEX と共通定義を整備してから、マスター計画の検索・招待の文言変更を行うと、二重修正を防げる。
